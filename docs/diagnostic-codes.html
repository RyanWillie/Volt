Volt validation contract

Diagnostic codes are part of the user-facing model.

Diagnostics describe bad design intent without making the kernel state invalid. Codes should be stable, documented, and structured enough for Python, future UI, tests, and automation.

Rule of thumb

Kernel mutation

Reject structurally invalid operations: missing IDs, dangling references, duplicate net ownership, invalid selected-part mappings.

Validation

Report bad circuit intent: unconnected required pins, single-pin nets, power-domain mistakes, overloaded selected parts.

Entity refs

References explain diagnostics. They must not become traversal or mutation handles in normal code paths.

Current codes

Stable code catalog

Code Severity Meaning Entity reference order
UNCONNECTED_REQUIRED_PIN Error A required pin is not connected to any net. pin, component, pin_definition
PIN_MUST_NOT_CONNECT Error A pin marked no-connect or must-not-connect is connected. pin, component, pin_definition, net
EMPTY_NET Warning A net exists with no connected pins. net
SINGLE_PIN_NET Warning A net has only one connected pin. net, pin
SELECTED_PART_VOLTAGE_RATING_EXCEEDED Error A net voltage exceeds a connected component selected-part voltage rating. net, pin, component
PIN_VOLTAGE_RANGE_VIOLATION Error A net voltage is outside a connected pin definition voltage range. net, pin, pin_definition
POWER_INPUT_WITHOUT_SOURCE Error A typed power input is connected to a non-ground net with no typed power source. net, then each consuming pin
PIN_GROUND_ON_NON_GROUND_NET Error A typed ground pin is connected to a net that is not marked ground. net, pin, pin_definition
PIN_POWER_ON_GROUND_NET Error A typed power pin is connected to a ground net. net, pin, pin_definition
MULTIPLE_OUTPUTS_ON_NET Error A net has multiple output drivers. net, then each output pin
PHYSICAL_PART_REQUIRED Error A component is missing the selected physical part required for PCB readiness. component, component_definition
Adding codes

Guidelines for future diagnostics

  • Use uppercase snake-case codes and treat them as stable once released.
  • Prefer one clear cause per diagnostic. Do not fold unrelated failures into one broad code.
  • Use Error when the design is likely electrically wrong or cannot be implemented as intended.
  • Use Warning for suspicious but plausible design states.
  • Put the primary entity first, followed by the most useful supporting entities.
  • Keep checks based on kernel-owned model data. Python, UI, and importers should not own hidden ERC meaning.
Validation passes

Current pass layout

validate_circuit(circuit)
  validate_connectivity(circuit)
  validate_electrical_rules(circuit)

validate_for_pcb(circuit)
  validate_circuit(circuit)
  validate physical part selection

Use validate_circuit(circuit) for the default complete logical validation suite. The named entry points expose stable intent-level groups for tests and future tools. Use validate_for_pcb(circuit) when a workflow needs physical implementation readiness; the smaller pass helpers remain internal structure.

This catalog should evolve with validation. When a PR adds a new diagnostic code, update this page and add a focused test for the emitted code, severity, and entity references.
