Planning guide - 29 May 2026

PCB should be CAD-like to author, but kernel-owned and netlist-safe underneath.

Scope: first PCB foundation, API shape, data model, staged development.

Position: footprints belong to the part/footprint library, not the board authoring API.

Outcome: a board projection that places selected physical parts and resolves pads to existing nets.

Core correction

The board API should not select footprints.

A logical component or selected physical part should already carry the package, footprint reference, and pin-pad mapping. If that data is absent or cannot be resolved to footprint geometry, PCB placement should fail or validation should report it. The board API consumes physical-part data; it does not patch it in.

Intended outcome
Author

CAD-like board shape

Define outlines, holes, slots, cutouts, fiducials, keepouts, and mechanical features with local coordinate frames and relative construction helpers.

Place

Schematic-like layout

Place existing logical components using absolute coordinates when needed and relative placement helpers for ergonomic layout.

Verify

Pad-to-net truth

Resolve every footprint pad back to logical pins and existing nets, then diagnose missing footprints, missing placements, invalid mappings, and unresolved pads.

Guiding principles

General primitives first.

The PCB layer should be built from a small vocabulary of reusable physical primitives: board, layer, outline, footprint, pad, placement, feature, copper, via, zone, rule, and diagnostic. Avoid circuit-specific kernel APIs such as USB-board helpers, regulator-placement APIs, or resistor-specific board objects. Those can exist as Python conveniences later, but they must lower into the same generic primitives.

1

PCB implements existing connectivity.

Only the logical circuit creates nets. PCB pads, tracks, vias, zones, and ratsnest views reference existing NetId values; they never create, merge, split, or rename nets.

2

Footprint choice belongs to parts.

The board API places components. It does not select footprints. Footprint identity, package, and pin-pad mapping come from selected physical parts and part/footprint libraries.

3

Authoring sugar lowers to stored primitives.

CAD-like builders and relative placement helpers are authoring conveniences. The kernel stores normalized geometry, placements, layers, features, and references.

4

Mechanical and electrical intent stay distinct.

Mounting holes, slots, fiducials, and cutouts are board features unless they carry electrical intent. Test points, connectors, chassis lugs, and tied mounting holes should be logical components when they participate in the circuit.

5

Derived truth is not duplicated.

Pad-to-net resolution, ratsnest lines, unrouted-net lists, and clearance findings should be derived from canonical board and circuit data, not stored as competing sources of truth.

6

Invalid state is rejected; bad layout is diagnosed.

Dangling IDs, invalid layers, impossible pad geometry, and malformed outlines fail at mutation boundaries. Unplaced components, unresolved footprints, unrouted nets, overlaps, and rule violations are diagnostics.

7

Persist Volt data before adapter data.

Deterministic Volt PCB JSON is the source of truth. KiCad PCB, Gerber, Excellon, STEP, and other formats should be import/export adapters over the kernel model.

8

Progressive disclosure.

Simple boards should need only an outline and placements. Advanced users can add explicit stackups, keepouts, rule classes, copper, zones, manufacturing details, and PCB-aware simulation later.

9

Units and coordinate frames are explicit.

Store physical lengths, angles, transforms, sides, and layer references explicitly. Relative positioning APIs may compute values, but stored PCB data should be deterministic and inspectable.

API feel

Use CAD patterns for geometry and Volt patterns for electronics.

Board construction can feel like Build123D or CadQuery: scoped builders, local frames, anchors, offsets, and shape operations. Component placement should feel closer to the schematic API: relative positioning, alignment, regions, and named anchors. Both lower into deterministic kernel data.

board = design.board("Main", size=("50mm", "35mm"))
board.layers.two_layer(thickness="1.6mm")

with board.outline() as outline:
    outline.rectangle("50mm", "35mm").fillet("2mm")
    outline.mounting_hole("M1", at=outline.nw.offset("4mm", "-4mm"), drill="3.2mm")

u1 = board.place(U1).at(board.center)
c1 = board.place(C1).near(u1.pad("VDD"), distance="2mm").align_y(u1)
r1 = board.place(R1).right_of(D1, gap="2mm")
j1 = board.place(J1).on_edge("left").align_y(board.center)

report = board.validate()
board.preview_svg("main-board.svg")

Important

This API assumes U1, C1, R1, and J1 already have selected physical parts with resolvable footprints. If they do not, placement should fail with a direct diagnostic.

Core data structures

Board

Project-owned PCB projection over one logical circuit. Stores metadata, units, layer stack, outline, features, placements, and later copper/rules.

LayerStack / BoardLayer

Named layers with roles: copper, solder mask, paste, silkscreen, fabrication, edge cuts, drill, mechanical, courtyard, keepout.

BoardOutline

Closed 2D boundary made from lines/arcs or normalized polygons. Defines the manufacturable board shape.

BoardFeature

Physical-only features: mounting holes, slots, cutouts, fiducials, tooling holes, text, mechanical keepouts.

FootprintDefinition

Normalized footprint geometry resolved from a library reference. Contains pads, holes, graphics, courtyard, and body/fab outlines.

FootprintPad

Stable pad identity, user label, kind, shape, local position, size, layers, drill, plating, and optional thermal/mechanical role.

ComponentPlacement

References an existing ComponentId, stores resolved footprint identity, position, rotation, side, and lock/status metadata.

PadResolution

Derived table linking placement pad to selected-part pin-pad mapping, concrete PinId, and existing NetId.

How data links
Logical ComponentId
  -> Component.selected_physical_part
  -> PhysicalPart.footprint_ref
  -> FootprintLibrary.resolve(footprint_ref)
  -> FootprintDefinition.pads
  -> PhysicalPart.pin_pad_mappings
  -> concrete PinId
  -> existing NetId
  -> PCB PadResolution

The resolver is the center of PCB correctness.

Placement only says where a selected part sits. Pad resolution proves what each physical pad means electrically. A pad is either tied to an existing net, intentionally non-electrical, intentionally unconnected, or invalid.

Development stages
1

PCB architecture and API agreement

Define ownership, value types, mutation boundaries, and the first API surface. Update Linear scope if older tickets imply board-level footprint selection.

Outcome: agreed model and acceptance criteria before implementation.

2

Footprint readiness in the part/library layer

Make selected physical parts resolve to normalized footprint definitions. Add a tiny built-in footprint set for simple examples: 0603 resistor, 0603 LED, pin header, small IC/package.

Outcome: board placement can error cleanly when physical data is missing.

3

Board geometry and CAD-like authoring

Add board outline, coordinate frames, layer stack, and physical board features. Python authoring can use context builders and relative anchors, but stores normalized kernel geometry.

Outcome: a board can exist before any components are placed.

4

Component placement and pad resolution

Place existing logical components with selected physical parts. Resolve pads to pins and nets. Reject dangling IDs and missing footprint data; diagnose design completeness issues.

Outcome: placement-only PCB model with truthful pad-to-net mapping.

5

Persistence, validation, and preview

Add deterministic PCB JSON, round-trip tests, PCB readiness diagnostics, and an SVG placement preview showing outline, features, footprints, pads, references, and optionally ratsnest lines.

Outcome: reviewable board artifacts.

6

Copper primitives after placement

Add tracks, vias, and later zones that reference existing NetId values. Start with manual routing and simple DRC, not autorouting.

Outcome: a manually routable board model.

Validation policy

Reject at mutation

  • Placing an unknown component.
  • Referencing an unknown footprint definition.
  • Creating a pad with invalid geometry or layer set.
  • Creating an outline that is not closed.
  • Creating copper on an unknown layer or net.

Diagnose as board design

  • Component has no selected physical part.
  • Selected part footprint cannot be resolved.
  • Component is unplaced.
  • Pad mapping misses a required logical pin.
  • Footprint is outside the board outline.
  • Net is unrouted or copper violates rules.
Product viewer fit

Deterministic PCB JSON should be product-viewer ready.

The first PCB JSON format should be useful beyond save/load. It should be a stable semantic board representation that a future 2D or 3D product viewer can consume directly, while keeping Python/code and the kernel model as the source of truth.

Identity

Stable selectable entities

Boards, placements, pads, features, tracks, vias, zones, and diagnostics need stable IDs so a viewer can select, highlight, inspect, and report against the same kernel-owned objects.

Geometry

3D-friendly primitives

Store outlines, thickness, layers, holes, pad geometry, package body outlines, component height, and optional 3D model references in a form that can generate meshes without reverse-engineering renderer output.

Inspection

Entity references everywhere

A clicked component should map to a placement and logical component. A clicked pad should map to placement, pad, pin, and net. A diagnostic should identify the board entities it concerns.

Immediate decision

Make the first implementation placement-only.

The first PCB slice should prove board geometry, footprint resolution, component placement, pad-to-net mapping, validation, persistence, and preview. Tracks, vias, zones, DRC, KiCad PCB export, Gerbers, and autorouting should follow only after the placement model is stable.
