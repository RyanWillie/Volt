Schematic PR 9

KiCad is an adapter, not Volt's schematic model.

Volt now has the canonical pieces a KiCad adapter should target: logical circuits, selected physical parts, schematic symbols, sheets, symbol placement, wire runs, net labels, JSON persistence, Python authoring, and SVG output. The next step is to define how KiCad import/export lowers into those structures without letting KiCad become the internal architecture.

Thesis
KiCad files are interoperability documents. Volt data is the source of truth once imported, and Volt's kernel-owned logical and schematic models remain the only place EDA meaning lives.

What this slice decides

  • How KiCad schematic and symbol data maps into Volt structures.
  • Which KiCad constructs are accepted, deferred, or reported as loss.
  • How export should consume Volt data without re-inferring a netlist.

What this slice does not implement

  • No KiCad parser.
  • No KiCad writer.
  • No schematic auto-layout.
  • No ERC, simulation, or PCB routing work.
Architecture rule

Import may create Volt objects; projection may not create connectivity.

A KiCad import is allowed to build a logical circuit by calling the logical circuit mutation APIs. After that, imported schematic data must reference the resulting ComponentId and NetId values. A wire drawn in KiCad can help the adapter determine the logical net during import, but once lowered into Volt it becomes a WireRun over a canonical net, not a second connection graph. 

KiCad file
  -> adapter-local parsed S-expression nodes
  -> Circuit mutations for components, pins, nets, selected parts
  -> Schematic mutations for sheets, symbols, placement, wires, labels
  -> deterministic Volt logical JSON + Volt schematic JSON
This keeps import powerful without weakening the projection boundary. The adapter can infer connectivity while importing a foreign file, but the stored Volt schematic never owns that connectivity.
Import pipeline
1. Parse Read .kicad_sch and .kicad_sym into adapter-local S-expression structures. Do not expose these structures through Volt core.
2. Resolve Resolve library identifiers, references, pin names, pin numbers, labels, footprint fields, and sheet paths into an import context.
3. Lower logic Create or match ComponentDefinition, component instances, pins, nets, and selected physical part data through kernel APIs.
4. Lower schematic Create SymbolDefinition, sheets, symbol instances, wire runs, and net labels over existing logical IDs.
5. Report loss Return warnings for unsupported constructs, generated names, hierarchy gaps, or footprint-only data that cannot safely become a selected part.
Export pipeline

Export should be boring: Volt in, KiCad out.

KiCad export consumes the same canonical data used by SVG rendering and JSON persistence. It should not look at wire geometry to rediscover what connects to what. It should write KiCad symbols, schematic symbol instances, wires, labels, and properties from Volt tables. 

Circuit + Schematic
  -> KiCad symbol definitions from SymbolDefinition
  -> KiCad schematic symbols from SymbolInstance + ComponentInstance
  -> KiCad wires from WireRun
  -> KiCad labels from NetLabel + canonical NetName
  -> KiCad properties from selected parts and safe provenance metadata
Export-first implementation is the safer first code slice after this design: it proves Volt can speak KiCad while keeping Volt as the source system. Import can follow once the mapping and loss-report shape are explicit.
Mapping table
KiCad construct Volt target Rule
.kicad_sym symbol SymbolDefinition Import supported drawing primitives and pin anchors into Volt's structured symbol model.
KiCad symbol pin SymbolPin plus logical PinDef match Match by stable pin number first, then name if needed. Mismatches become import errors or warnings.
Schematic symbol instance ComponentInstance and SymbolInstance The component owns identity and reference designator. The symbol instance owns placement only.
KiCad reference/value/properties Component reference and safe property metadata Known fields lower into typed Volt fields; unknown fields stay metadata.
KiCad footprint field PhysicalPart only when complete Do not invent a manufacturer part from a footprint string. Preserve footprint-only data as provenance until Volt has a footprint-only assignment primitive.
KiCad wires WireRun over NetId During import, wires help resolve nets. In Volt storage, they reference existing nets.
Local/global labels NetName plus NetLabel Labels create or resolve logical net names during import; schematic labels render the canonical net name afterward.
Power symbols Named nets and optional typed net attributes Power symbols are import syntax. They do not create a special power-rail object by themselves.
No-connect marker Future explicit no-connect assertion Report as unsupported until Volt owns this design-intent primitive.
Hierarchical sheets Future module-aware schematic projection Reject or warn initially. Do not flatten silently if it changes meaning.
Compatibility snapshot

The current KiCad boundary is a narrow flat-schematic subset.

This matrix describes the supported adapter contract for the first KiCad schematic path. It is intentionally a compatibility boundary, not a promise that Volt stores KiCad's full file model internally. On main as of VOL-65, the adapter target and loss-report vocabulary exist, while the flat .kicad_sch writer and golden KiCad fixtures are still expected from VOL-63. Until those fixtures land, fixture-backed evidence comes from Volt's canonical logical and schematic fixtures plus the KiCad boundary tests.

Construct Supported subset Fixture or test evidence
Symbol primitives Lines, rectangles, circles, arcs, and text are the supported Volt symbol primitives that a KiCad writer may emit. KiCad stroke, fill, font, and graphical styling beyond those structured primitives is lossy unless Volt later owns equivalent presentation data. schematic_writer_test.cpp and schematic_svg_writer_test.cpp
Symbol pins Pin name, pin number, anchor, and orientation are supported where they map to SymbolPin. Pin electrical meaning remains owned by Volt logical pin definitions, not the KiCad drawing. schematic_test.cpp and schematic_reader_test.cpp
Symbol instances A placed symbol may carry its symbol definition, component reference, sheet, position, and orientation. It must point at an existing Volt component; it must not create connectivity by itself. schematic_writer_test.cpp and STM32 schematic fixture
Wires Wire geometry is supported as WireRun presentation over an existing NetId. Export must use the canonical net attached to the wire run; it must not rediscover a netlist from geometry. schematic_test.cpp and schematic_writer_test.cpp
Labels Local net labels are supported when they render a canonical Volt net name at a schematic point. Global labels, hierarchical labels, and sheet pins are deferred until Volt owns the matching hierarchy semantics. schematic_writer_test.cpp and stm32_usb_buck/schematic_output.py
References, values, and properties Component references are supported through Volt reference designators. Values and safe user properties may be emitted from Volt component or selected-part metadata when they do not imply new EDA meaning. Unknown KiCad fields should remain provenance or produce a loss warning. parts_test.cpp, logical_circuit_writer_test.cpp, and led_circuit.volt.json
Footprint fields A complete Volt selected physical part may carry a footprint library and name. A bare KiCad footprint string is incomplete selected-part data; it should be preserved as provenance or reported as incomplete until Volt owns a footprint-only assignment primitive. led_circuit_test.cpp and typed_electrical_attributes.volt.json
When VOL-63 lands the flat writer, this section should link directly to the golden .kicad_sch fixture. Until then it is a pre-export boundary snapshot grounded in Volt's existing schematic and logical fixtures.
Unsupported and lossy

Unsupported KiCad constructs are reported; they are not smuggled into core.

The adapter reports imperfect interoperability through LossReport. A warning has a severity, a category, the KiCad construct name, and a human-readable message. Import/export callers can render those warnings in Python, CLI output, or structured logs without giving KiCad file-format details ownership over Volt's kernel model.

KiCad construct Boundary behavior Report category
No-connect markers Reject or warn until Volt has an explicit no-connect design-intent primitive. Do not model them as missing wires or hidden properties. UnsupportedConstruct
Hierarchical sheets, sheet pins, hierarchical labels Defer until module-aware schematic projection can preserve meaning. Do not flatten silently when it would change connectivity or scope. UnsupportedConstruct
Buses and bus entries Defer until Volt owns bus semantics. Individual wire runs may only be exported when they already refer to canonical scalar Volt nets. UnsupportedConstruct
Junction dots, graphical connection hints Import may use them while lowering a foreign file into logical nets. Stored Volt schematic data must reference canonical nets afterward. LossyConstruct if presentation is approximated
Rich symbol styling Stroke details, fill details, fonts, text effects, and KiCad-only graphical flags are approximated or omitted unless represented by Volt symbol primitives. LossyConstruct
Footprint-only assignments Preserve as provenance or warn until Volt has a kernel-owned footprint-assignment concept. Do not create incomplete selected parts. IncompleteConstruct
PCB, routing, ERC, simulation, project metadata Out of scope for the schematic adapter. These belong to future explicit Volt model data or validation layers, not to the KiCad schematic path. UnsupportedConstruct
Missing compatibility coverage should become a follow-up ticket such as VOL-63 fixture linkage or a future hierarchy/bus/no-connect ticket. VOL-65 should not grow the KiCad parser or writer behavior.

Boundary tests: kicad_boundary_test.cpp keeps KiCad adapter headers outside the Volt umbrella, and kicad_loss_report_test.cpp fixes the structured warning vocabulary used by unsupported, incomplete, and lossy constructs.

Key decisions
Decision 1

Adapter-local KiCad AST only.

KiCad S-expression structures belong under an adapter namespace or target. Core schematic, circuit, authoring, and validation targets must not depend on KiCad parser types.

Decision 2

Do not fake selected physical parts.

A KiCad footprint reference is useful, but it is not automatically a manufacturer part. Populate PhysicalPart only when the source data is complete enough to satisfy the kernel's selected-part invariants.

Decision 3

Use a loss report for imperfect interoperability.

Import/export should return a structured report with warnings for unsupported fields, generated identifiers, graphical details Volt does not model, and deferred design-intent features.

Decision 3a

Return warnings through adapter results.

Python and CLI callers should receive the same KiCad adapter loss report: each warning carries a severity, a category such as unsupported, incomplete, or lossy, the KiCad construct name, and a human-readable message. CLI commands can render those records to stderr or JSON, while Python bindings can expose the records as structured objects next to the imported or exported Volt data.

Decision 4

Export flat schematics before importing complex KiCad projects.

Exporting Volt's own flat schematic model to KiCad is the smallest proof of interoperability. Import, hierarchy, and richer library compatibility should follow after the first writer is deterministic.

Later

Acceptance

Architecture preserved

KiCad code depends on Volt circuit and schematic structures. Volt core does not depend on KiCad file-format types.

Mapping is explicit

Components, symbols, selected parts, sheets, placements, wires, and labels all have a documented target or a documented deferral.

No hidden netlist

Export uses canonical Volt nets. Import may infer nets only while creating the logical circuit, before schematic projection objects are stored.

Hard no
Do not make KiCad's file schema Volt's internal schematic schema. Do not store raw KiCad S-expressions as the only source of truth. Do not let schematic wires become a second connectivity model.

Not now

  • Full KiCad project import.
  • Hierarchical sheet import/export.
  • Bus semantics.
  • No-connect design-intent primitives.

Still separate

  • PCB placement and routing.
  • ERC and simulation.
  • BOM generation and supplier logic.
  • Automatic schematic layout.
Notes: KiCad documents modern schematic files as .kicad_sch and symbol libraries as .kicad_sym, both using KiCad's S-expression format. The KiCad docs also ask third-party writers to use their own generator identifier rather than pretending to be KiCad. Sources: schematic file format, symbol library file format.
