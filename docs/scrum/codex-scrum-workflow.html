Codex Scrum for Volt

A lightweight operating model for using Codex as a Scrum system around GitHub Issues, GitHub Projects, pull requests, and repo-local skills. The model instantiates Product Owner, Developer/Tech Lead, and Scrum Master responsibilities on demand while keeping durable state in GitHub, docs, and skills.

Role Boundaries

Product Owner

Owns direction, value, priority, and acceptance. The PO decides why work matters, what outcome is wanted, and what should come next.

  • Shapes backlog items around user or project value.
  • Orders work by roadmap impact, dependency pressure, and risk.
  • Rejects work that is technically interesting but not product-relevant.

Developer / Tech Lead

Owns architecture, feasibility, decomposition, and verification. The developer makes sure tasks are implementable without weakening Volt's kernel boundaries.

  • Checks layer ownership and invariants before implementation.
  • Splits work into scoped PR-sized technical slices.
  • Defines test and CI evidence for done work.

Scrum Master

Owns ceremony quality, flow, blockers, and improvement. The Scrum Master keeps the process honest and makes sure the workflow tightens after each sprint.

  • Maintains Definition of Ready and Definition of Done.
  • Tracks blockers, dependencies, and stale work.
  • Turns retrospectives into concrete process changes.

Operating Loop

Backlog Refinement

Use volt-backlog-refinement. Codex reads open GitHub issues carrying roadmap, docs, recent PRs, and architecture notes. The PO clarifies value and priority. The Developer tightens technical shape. The Scrum Master produces ready/not-ready outcomes and proposed issue or Project edits.

Sprint Planning

Use volt-sprint-planning. Codex selects a coherent sprint goal and candidate tickets, checks dependencies, and defines branch strategy, expected verification, and capacity risks.

Sprint Kickoff

Use volt-sprint-kickoff. Codex creates isolated branches or worktrees, dispatches developer agents only for independent work, and keeps GitHub issue, Project, branch, and pull request state synchronized.

PR Thermonuclear Review

After each PR is raised, Codex creates a fresh thread to run thermo-nuclear-code-quality-review. Findings, or a clean-review result, are recorded as a PR comment; the authoring agent then addresses actionable findings, pushes fixes to the peer branch, and comments with resolution evidence before the lane is review-ready.

Sprint Review

Use volt-sprint-review. Codex summarizes what shipped, what changed architecturally, open PR or CI state, unfinished work, and the resulting product capability.

Retrospective

Use volt-retrospective. Codex reviews tickets, PRs, CI, and conversation friction. It proposes updates to skills, AGENTS.md, docs, or ceremony checklists when the same problem repeats.

Agile Ticket Standard

Ticket Shape

Every sprint candidate should read as a small promise of value, not just a task title. The Product Owner owns the outcome and acceptance criteria; the Developer owns the technical approach and verification notes.

  • Problem, user/project value, and intended outcome.
  • Acceptance criteria written as observable results.
  • Non-goals, dependencies, blockers, and related tickets.
  • Technical notes for architecture fit, touched surfaces, and verification.

Definition of Ready

A ticket is ready when it can enter a sprint without hidden product decisions or undefined technical ownership. Not-ready work stays in refinement, even if it is important.

  • Value, priority, and sprint-goal fit are clear.
  • Acceptance criteria and non-goals are testable.
  • Dependencies and blockers are named.
  • Scope is small enough for one PR or an explicit sequence.

Definition of Done

Done means the increment is usable, reviewed, and traceable. A merged PR alone is not enough if acceptance, verification, GitHub issue hygiene, or follow-up capture is missing.

  • Acceptance criteria are met and out-of-scope work is captured separately.
  • Volt architecture boundaries and kernel-owned semantics are preserved.
  • Relevant tests, docs, local checks, and CI are passing or explicitly waived.
  • PR, thermonuclear review comment, author resolution evidence, GitHub issue status, and follow-up issues are complete.

Guardrails

The Product Owner does not decide architecture. The Developer does not reorder product priorities without explaining the technical constraint. The Scrum Master does not invent product scope. Codex may orchestrate all three roles, but every output must preserve these boundaries.

Durable memory belongs in GitHub Issues, GitHub Projects, pull requests, repo docs, and skills. Subagents are temporary workers; they should receive scoped prompts and report back with evidence.
