Volt Codebase Growth Optimization Audit

The growth is partly justified by real product surface area, but several areas are now growing faster than their ownership boundaries. The highest-leverage optimization is not mechanical line deletion; it is moving endpoint, pad, footprint, and rule semantics back behind kernel-owned typed APIs so Python can stay ergonomic instead of becoming a parallel EDA engine.

Priority Findings

01
High

Python schematic authoring is becoming a parallel semantic layer

python/volt/schematic.py is a 3,714-line facade that handles wire builders, drawing state, regions, two-terminal placement, labels, stubs, ports, IO, errors, and endpoint conversion. The risk is not only file size: methods around connect, _connect_existing_net, and _add_wire_with_endpoint_payloads normalize anchors, infer endpoint payloads, validate cross-design handles, and choose route intent before calling the kernel.

Evidence: python/volt/schematic.py:1738, python/volt/schematic.py:1866, and kernel-side endpoint resolution around src/schematic/schematic.cpp:642 and src/schematic/schematic.cpp:692.

Optimization: introduce kernel-facing endpoint request types such as SchematicEndpointRef and SchematicWireRequest. Let Python parse ergonomic calling forms, but pass typed endpoint refs into C++ for net resolution and structural checks. After that boundary exists, split schematic.py into focused authoring modules with the facade only re-exporting stable API.

02
High

PCB authoring infers pad/net identity in Python

python/volt/_pcb_composition.py and python/volt/_pcb_layout.py infer routing net identity from pad anchors, while C++ already owns pad resolution and board routing. This is exactly the kind of EDA meaning the project guidelines say should live in the kernel.

Evidence: python/volt/_pcb_composition.py:156, python/volt/_pcb_composition.py:378, python/volt/_pcb_layout.py:523, with kernel ownership in src/pcb/board.cpp:158 and src/pcb/board_router.cpp:79.

Optimization: expose a BoardEndpoint or route_between_endpoints API that accepts pad/point endpoint refs and lets C++ resolve the net. Python can keep layout.connect(start, end) sugar, but should not decide whether two endpoints share a net.

03
High

Generated example artifacts are tracked in duplicate forms

The repository tracks 57 generated example artifact files totaling 18,807 lines. Several outputs are duplicated in both legacy flat paths and newer project-package paths, including SVG, JSON, validation, and KiCad outputs. Golden files are useful, but the current shape makes diffs look larger than the source change that produced them.

Evidence: examples/*/artifacts/** includes duplicated flat outputs and nested .volt/ package outputs for timer, PCB LED, and STM32 examples.

Optimization: choose one golden-output policy. Prefer keeping only canonical package outputs plus small targeted fixtures for tests. Generate broad visual/export artifacts in CI or release jobs unless a test asserts that exact file. This can remove thousands of reviewed lines without weakening behavior coverage.

04
Medium

Three JSON readers carry their own mini-frameworks

logical_circuit_reader.cpp, schematic_reader.cpp, and pcb_reader.cpp each define local helpers for require, field lookup, typed fields, optional fields, local IDs, version checks, and reference resolution. That duplication is now large enough to create schema behavior drift.

Evidence: helper clusters begin around src/io/logical_circuit_reader.cpp:37, src/io/schematic_reader.cpp:38, and src/io/pcb_reader.cpp:41.

Optimization: add src/io/detail/json_fields.hpp or a small JsonReaderContext for typed extraction and consistent error messages. Keep domain parsing local; centralize the repetitive structural reader mechanics.

05
Medium

Board polygon validation is duplicated almost line-for-line

BoardOutline and BoardPolygon repeat duplicate-closing-vertex cleanup, signed area, segment intersection, and self-intersection validation. The meaningful difference is mostly the noun in error messages and BoardOutline::contains.

Evidence: src/pcb/board_outline.cpp:59 through src/pcb/board_outline.cpp:239.

Optimization: use one internal validation path over std::vector<BoardPoint> with a context noun, or a private validated polygon helper. Keep public types separate; delete the repeated geometry mechanics.

06
Medium

Schematic and PCB Python anchors repeat the same mechanics

SchematicAnchor and BoardAnchor share x/y/point access, offsets, directional moves, axis targeting, iteration, and repr. Placed schematic and placed board components also share pin lookup, ambiguity handling, and dynamic attribute access patterns.

Evidence: python/volt/_schematic_handles.py:16 and python/volt/_pcb_layout.py:35.

Optimization: extract a tiny internal 2D anchor base/factory and a shared pin-anchor lookup helper. Keep schematic and PCB types distinct at the API boundary so domain semantics do not blur.

07
Medium

Enum schema conversion is hand-rolled repeatedly

schematic_schema.cpp and pcb_schema.cpp use repeated switch and string-if chains for enum names and parsing. This is verbose and easy to desynchronize when schemas evolve.

Evidence: repeated enum conversion patterns start near src/io/schematic_schema.cpp:5 and src/io/pcb_schema.cpp:14.

Optimization: use small constexpr mapping tables plus generic enum_name and enum_from_name helpers. This is a safe local reduction because schema ownership remains in C++.

08
Medium

Physical footprint/model caches are split across Python and kernel state

Python carries object footprint caches, model asset maps, cached board footprints, and dataclasses mirroring footprint/part structures. Some of this is packaging concern, but geometry and mapping identity should not drift from kernel-owned selected physical parts.

Evidence: python/volt/design.py:34, python/volt/logical.py:534, python/volt/library.py:170, python/volt/pcb.py:195.

Optimization: route object-authored footprint definitions through a kernel-owned footprint registry/cache keyed by kernel IDs. Keep Python asset-copy bookkeeping, but move geometry identity and selected-part mappings into C++.

Recommended Sequence

First: stop review-noise growth

  1. Decide the generated artifact policy for examples/**/artifacts.
  2. Keep only canonical golden outputs required by tests.
  3. Add a check that prevents accidental committed dist/, __pycache__, and regenerated duplicate artifacts.

Second: move semantics to the kernel

  1. Add schematic endpoint request types and binding helpers.
  2. Add PCB endpoint routing APIs that resolve net identity in C++.
  3. Move board rule/capability normalization out of Python policy maps.

Third: delete repeated mechanics

  1. Centralize JSON reader field extraction.
  2. Centralize board polygon validation.
  3. Use enum mapping tables for schema conversions.

Fourth: decompose large facades

  1. Split schematic.py after endpoint semantics move down.
  2. Keep public imports stable from python/volt/schematic.py.
  3. Split by ownership boundary: endpoints, routes, ports, drawing, IO.

What Looks Justified

  • The C++ circuit kernel largely follows the project principle that invalid kernel state should be impossible; core connectivity checks still live in mutation APIs.
  • Tests are large, but much of that size buys important structural and round-trip coverage for a format-heavy EDA kernel.
  • Documentation growth is substantial but useful while the architecture is still forming; the concern is only when docs mask unresolved ownership decisions.
Investigation used local file metrics, tracked-file inspection, targeted source reads, and two scoped subagent audits. No implementation files were changed.
