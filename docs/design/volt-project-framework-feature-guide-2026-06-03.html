Feature guide

Volt Project Framework

A structured, testable, and flexible way to create electronics with Python while keeping EDA meaning in the Volt kernel.

Date: 2026-06-03

Status: Planning guide

Audience: Volt maintainers, future agents, application builders, and component-library authors.

Purpose

The framework gives Volt Python projects a real shape.

Today a user can define electronics in Python, but a serious product needs more than a script that happens to construct a design. It needs a known build entrypoint, reusable parts, libraries that can be tested, multiple boards, rules, diagnostics, assets, simulation scenarios, and deterministic outputs that an application or agent can load.

The Volt Project Framework should be the boring structural layer that turns Python electronics authoring into a repeatable engineering workflow. It should not become a package manager, task runner, application runtime, CAD kernel, or alternate EDA database. It should simply make it obvious where a design is built, which libraries and rules are used, which boards belong to the product, what artifacts were produced, and whether the result is valid.

Goal

Let users define real electronics in Python with enough structure that the same source can be checked, tested, viewed, simulated, exported, and modified by an agent.

Non-goal

Do not make Project the owner of circuit semantics. The kernel remains the source of truth for connectivity, selected parts, PCB data, rules, and diagnostics.

Principles

Structured, not heavy

Project gives the codebase a known shape. It should feel like a small harness around electronics authoring, not like adopting a platform.

Kernel-owned truth

Python provides ergonomic construction. The kernel owns EDA meaning: components, pins, nets, schematic presentation, board placement, routing, rules, and validation.

Parts first

A user should be able to say "here is a part." A part can include pins, symbol, footprint, pad mapping, physical metadata, 3D representation, ratings, and optional simulation behavior.

Libraries are buildable

A library is a collection of parts that can be imported by projects and tested independently. A library should prove its parts are board-ready before a project depends on them.

Products can be multi-board

A project should be able to contain one board, many boards, fixtures, harnesses, variants, and system-level relationships without turning those into random Python globals.

Deterministic outputs

Running the same project should produce stable artifacts: logical JSON, schematic data, PCB JSON, diagnostics, assets, simulation inputs, and manufacturing outputs when available.

General primitives

Prefer small composable primitives over narrow circuit-specific methods. The framework should orchestrate those primitives rather than grow a large special-case API surface.

Core Model

Part

Reusable definition of an electronic part and its usable physical representations.

Library

Buildable collection of parts with its own validation and tests.

Project

Known entrypoint that assembles designs, boards, rules, assets, and outputs.

Kernel

Owns the actual EDA state and rejects invalid structural mutations.

Result

Deterministic bundle for apps, agents, tests, simulation, and export.

Concept Owns Does not own
Part Reusable part definition: logical pins, symbol, footprint, pad mapping, physical metadata, optional 3D and simulation declarations. A placed component instance, board coordinates, or project-specific net connections.
Library A named collection of parts plus tests and library-level validation. Project build orchestration, selected product variants, or board layout.
Design The logical circuit: component instances, pins, nets, hierarchy, selected parts, and circuit diagnostics. Manufacturing files, viewer bundle shape, or application workflow state.
Board The physical PCB implementation of a design: outline, layers, placements, routes, copper, holes, cutouts, text, keepouts, and board rules. The netlist itself. PCB implements logical connectivity; it does not define it.
Project The build entrypoint, build context, product-level organization, rule composition, artifact collection, asset collection, and result emission. EDA semantics. It asks the kernel to build, validate, and serialize the source-of-truth models.
ProjectResult The output of one run: models, artifacts, diagnostics, assets, metadata, and pass/fail status. Long-lived history, caching policy, package management, or source control.

User Experience

The public API should make the simple case feel almost as direct as plain Python, while giving larger products a natural place to grow. A user should not need to learn a build system before making a circuit. They should only need to name a project and provide a build function.

import volt
from volt_components.passives import R_0603_10K
from volt_components.indicators import LED_0603_RED

project = volt.Project("status-led")

@project.build
def build(ctx):
    design = volt.Design("status-led")

    r1 = design.instantiate(R_0603_10K, ref="R1")
    d1 = design.instantiate(LED_0603_RED, ref="D1")

    design.connect(r1.pins["2"], d1.pins["A"])
    design.connect(d1.pins["K"], design.gnd)

    board = volt.Board("status-led-board", design=design)
    board.place(r1, at=volt.Position(12, 10))
    board.place(d1, at=r1.position.right(8))

    ctx.add(board)
    ctx.use_rules(volt.rules.erc.basic())
    ctx.use_rules(volt.rules.pcb.two_layer_manufacturable())

    return design

The build function is the contract.

The eventual application, CLI, CI job, or agent runner can discover the project, run the build function, and consume the result without guessing how the user's Python is organized.

Multi-Board Products

A real electronics product may have a main board, daughterboard, programming adapter, sensor module, front-panel board, cable harness, or test fixture. Without a project layer, these become loose files or unrelated Python variables. With a project layer, the product can be represented as one coherent build.

project = volt.Project("robot-controller")

@project.build
def build(ctx):
    controller = make_controller_design()
    sensor = make_sensor_design()
    panel = make_panel_design()

    controller_board = make_controller_board(controller)
    sensor_board = make_sensor_board(sensor)
    panel_board = make_panel_board(panel)

    system = volt.Assembly("robot-controller")
    system.add_board(controller_board)
    system.add_board(sensor_board)
    system.add_board(panel_board)
    system.connect(controller_board["J2"], sensor_board["J1"])
    system.connect(controller_board["J3"], panel_board["J1"])

    ctx.add(system)
    ctx.use_rules(product_rules())

    return system
What this enables
  • Product-level validation across several boards.
  • Shared libraries and rule sets.
  • Board-specific artifacts grouped under one result.
  • Agent workflows that understand the whole product.
Careful boundary

Assembly may describe product structure, connectors, harnesses, and relationships. It should not bypass the individual logical designs or PCB models owned by the kernel.

Rules And ERC

The project is the right place to select and compose rules because rules are part of the product context. A library can contribute part-level expectations. A design can be checked with ERC. A board can be checked with DRC and manufacturability constraints. A product can add higher-level rules such as "all external connectors require ESD protection" or "this variant must not use unavailable parts."

def product_rules():
    return volt.RuleSet(
        volt.rules.erc.basic(),
        volt.rules.erc.power_domains(),
        volt.rules.parts.require_selected_physical_parts(),
        volt.rules.pcb.clearance(minimum="0.15mm"),
        volt.rules.pcb.hole_to_copper(minimum="0.25mm"),
        volt.rules.product.external_connector_esd(),
    )

@project.build
def build(ctx):
    design = make_design()
    board = make_board(design)

    ctx.add(board)
    ctx.use_rules(product_rules())

    return design
Rule level Examples Selected by Evaluated by
Library Part has pins, footprint, pad mapping, ratings, model refs, simulation claims. Library build/test Kernel validation and library validators.
Design Unconnected pins, single-pin nets, incompatible electrical roles, power-domain mistakes. Project or default profile Kernel diagnostics.
Board Unplaced components, unresolved footprints, unrouted nets, clearance, holes, copper, stackup. Project, board profile, manufacturer profile Kernel diagnostics and PCB validators.
Product Multi-board connector compatibility, variants, part availability, test-point coverage, safety rules. Project Kernel-owned constraints where available, project-level orchestration otherwise.

Rule ownership matters.

The project chooses rule sets. The kernel should own rule semantics wherever the rule changes what Volt understands about the electronics. This keeps diagnostics inspectable and prevents Python-only correctness rules from becoming invisible to apps and agents.

Libraries

The framework should make official and private component libraries feel like first-class Volt citizens without turning Volt into a package manager. A library is simply a buildable collection of parts. A project imports those parts using normal Python import mechanisms.

import volt

library = volt.Library("volt_components.passives")

R_0603_10K = volt.Part(
    name="R_0603_10K",
    pins=[volt.PinSpec("1", 1), volt.PinSpec("2", 2)],
    footprint=volt.Footprint(
        library="Resistor_SMD",
        name="R_0603_1608Metric",
        pads=(
            volt.FootprintPad.surface_mount("1", at=(-0.75, 0), size=(0.8, 0.95)),
            volt.FootprintPad.surface_mount("2", at=(0.75, 0), size=(0.8, 0.95)),
        ),
    ),
    pads={1: "1", 2: "2"},
    value="10k",
    manufacturer="Yageo",
    mpn="RC0603FR-0710KL",
    package="0603",
    prefix="R",
)

library.add(R_0603_10K)
Importable

A project uses standard Python imports. Volt does not need to solve dependency distribution to structure electronics correctly.

Buildable

A library can produce a result that proves its parts are constructible, complete, and serializable.

Testable

Library tests can assert that parts are board-ready, simulation-ready, procurement-ready, or schematic-ready.

def test_passives_are_board_ready():
    result = library.build()

    assert result.ok
    assert result.part("R_0603_10K").has_footprint
    assert result.part("R_0603_10K").pad_mapping_complete
    assert result.part("R_0603_10K").serializable

Project Result

A project run should produce a ProjectResult. This is not a database, cache, history system, or source of truth. It is the result of running the user's project at one point in time.

result = project.run()

if not result.ok:
    for diagnostic in result.diagnostics:
        print(diagnostic)

result.write("dist/status-led.volt")

The result should be usable by the future Volt application, CLI, CI, tests, simulators, viewers, and manufacturing exporters. The app should not need to execute arbitrary Python just to inspect a finished board; it should load the project result bundle.

The bundle is the application contract.

For the future paid application, the project result is the stable handoff. The application can render the schematic, render the 3D board, select components, show diagnostics, request agent edits, run tests, and compare results without owning the kernel's EDA semantics.

Testing

The framework should make electronics tests feel natural. Users should be able to run small unit tests over parts, integration tests over designs, board-readiness tests over PCBs, regression tests over artifacts, and eventually simulation tests over scenarios.

Part tests

Does the part have complete pins, footprint, pad mappings, physical metadata, and optional simulation/model data?

Design tests

Does swapping a resistor, adding a subcircuit, or changing a variant preserve the intended connectivity and ERC status?

Board tests

Are all components placed, all footprints resolved, all required nets routed, and all board rules satisfied?

def test_status_led_project_builds_cleanly():
    result = project.run()

    assert result.ok
    assert result.diagnostics.count(severity="error") == 0
    assert result.board("status-led-board").all_required_nets_routed()


def test_led_current_path_survives_resistor_change():
    result = project.run(params={"led_resistor": "2.2k"})

    design = result.design("status-led")
    assert design.net_between("R1.2", "D1.A")
    assert result.diagnostics.count(code="ERC_FLOATING_INPUT") == 0

This is where Volt can become unusually strong.

Because electronics are built from Python into kernel-owned models, tests can operate on the actual circuit and board data. They do not need to scrape screenshots, parse generated netlists, or guess what a designer meant.

Simulation

The project framework gives simulation a better home. Simulation should not be a detached tool that asks the user to rebuild the circuit in another language. A project can declare scenarios over the same parts, nets, boards, and variants that define the product.

@project.simulation("button-press")
def button_press(ctx, design):
    ctx.set_supply("VCC", 3.3)
    ctx.drive("SW1.1", waveform=volt.waveforms.pulse(width="40ms"))
    ctx.probe("D1.A")
    ctx.probe("MCU.PA4")


def test_button_turns_led_on():
    result = project.run()
    sim = result.simulate("button-press", duration="100ms")

    assert sim.signal("D1.A").goes_high()
    assert sim.signal("MCU.PA4").observes_high()

The exact simulation engine can evolve. The framework's job is simpler: provide a deterministic place to define scenarios, bind them to the built design, collect simulation artifacts, and report diagnostics.

3D And Assets

The visualizer should not need to infer physical models from arbitrary Python code. Parts and boards should lower into assets and references that are written into the project result. The PCB remains primarily 2D/2.5D manufacturing data; the viewer gets enough structured board geometry, placement, stackup, and model references to render a true 3D scene.

Board JSON

Contains board outline, holes, cutouts, stackup, placements, footprint geometry, copper, and references to model assets.

Asset bundle

Contains generated or referenced model files, hashes, metadata, and paths that the viewer can load directly.

Agent Workflow

The framework gives an agent a stable loop. The agent edits Python, runs the project, reads the result, observes diagnostics and artifacts, and edits Python again. The application can provide richer feedback, but the framework makes the loop possible without baking application behavior into Volt core.

1

Agent edits Volt Python source.

2

Project build runs through the known entrypoint.

3

Kernel validates design, board, rules, and diagnostics.

4

Project result writes deterministic JSON and assets.

5

App, tests, viewer, or agent inspect the result.

This keeps the application free to innovate. It can offer chat, a code editor, a 3D viewer, component search, diagnostics panels, and specialist agents, while Volt remains the open kernel and Python framework underneath.

API Surface

The first API should stay small. It should establish the framework boundaries without committing Volt to every future workflow.

API Role First behavior
volt.Project(name) Defines the buildable project. Stores metadata, build function, params, rules, and output preferences.
@project.build Marks the canonical build entrypoint. Runs with a BuildContext and returns a design, board, assembly, or project object graph.
BuildContext Collects project-owned declarations. Allows ctx.add(...), ctx.use_rules(...), ctx.publish(...), ctx.asset(...), and diagnostics aggregation.
project.run(...) Executes one deterministic build. Builds kernel models, validates, serializes, collects artifacts and returns ProjectResult.
ProjectResult Represents the output of one run. Exposes models, artifacts, assets, diagnostics, status, and write(...).
volt.Library(name) Defines a reusable buildable part collection. Registers parts, validates them, and produces a library result for tests.
volt.Part(...) Defines a reusable component-level abstraction. Lowers into kernel-owned logical, schematic, PCB, model, ratings, and optional simulation data.

Keep the first version disciplined.

The first framework version does not need every future rule type, simulation engine, asset generator, or app hook. It needs the ownership boundaries, deterministic build lifecycle, result object, and enough artifact collection that future features naturally attach to it.

Implementation Plan

Phase 1

Project and ProjectResult foundation

Add volt.Project, BuildContext, ProjectResult, artifact registration, diagnostics aggregation, deterministic write layout, and tests for single-design/single-board projects.

Phase 2

Buildable libraries and parts

Add the first-class public Part and Library framework shape. Ensure libraries can validate board-ready parts independently and projects can instantiate imported parts without manual footprint registration.

Phase 3

Rules and diagnostics composition

Let projects select rule sets and emit structured diagnostics across design, schematic, PCB, library, and product-level checks. Keep rule semantics in the kernel or explicit model-owned validators.

Phase 4

Multi-board and product assembly

Add product-level grouping for multiple boards, shared variants, connectors, harnesses, and cross-board checks without weakening design and board ownership boundaries.

Phase 5

Application, viewer, simulation, and export bundle

Extend project results with viewer-ready PCB JSON, model asset references, simulation scenario artifacts, manufacturing outputs, and app-friendly manifests.

What This Unlocks

For users

A consistent way to define electronics in Python, build them, check them, and share the result without turning every project into bespoke script glue.

For agents

A reliable edit-run-inspect loop with deterministic artifacts and diagnostics instead of fragile assumptions about arbitrary source structure.

For apps

A clean contract for loading circuits, schematics, boards, assets, and diagnostics into a visual editor without owning the EDA model.

For tests

Regression checks over actual circuit and PCB data: part swaps, subcircuits, rules, board readiness, simulation behavior, and artifact stability.

For libraries

Official and private part libraries can be validated independently, then imported into projects as ordinary Python modules.

For Volt

A stronger foundation for simulation, PCB generation, 3D viewing, manufacturing export, and paid application workflows without compromising the open kernel.

Acceptance

The first complete version of this framework should meet these criteria:

  • A project has one discoverable build entrypoint and can be run by CLI, tests, or application code.
  • A project can return or register at least one logical design and one board.
  • A project can select rule sets and emit structured diagnostics.
  • A project run produces a deterministic ProjectResult with manifest, artifacts, diagnostics, and assets.
  • A library can define parts, validate them independently, and be imported by a project.
  • Parts can carry enough data to become schematic-ready, board-ready, viewer-ready, and eventually simulation-ready.
  • The framework does not duplicate kernel-owned EDA semantics in Python-only structures.
  • The result bundle is sufficient for an app or agent to inspect the current electronics state without executing arbitrary project code.

North star

A user should be able to build a real electronics product in Python, test it like software, inspect it like CAD, validate it like EDA, simulate it like a system, and hand the result to an application or manufacturing exporter. The Project Framework is the spine that makes those workflows feel like one Volt-native experience.
