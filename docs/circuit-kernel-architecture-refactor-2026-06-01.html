Circuit Kernel Architecture Refactor

One deliberate C++ kernel refactor: separate reads from writes, keep invalid kernel state impossible, and preserve the Python authoring API while allowing internal bindings to move.

2026-06-01 Scope: Volt C++ kernel architecture
Status: implementation plan

Decision

Keep
  • Circuit owns entity tables and structural invariants.
  • Structural invalidity throws at mutation boundaries.
  • Design-quality issues stay in diagnostics.
  • Python user-facing behavior stays compatible.
  • Serialization and golden fixtures stay deterministic.
Change
  • C++ call sites use CircuitView for reads.
  • Hierarchy, electrical, and design-intent writes move into cohesive kernel-owned modules.
  • Validation consumes CircuitView through deterministic rule suites.
  • Projection and adapter contracts become named, checked boundaries.

Module Layout

Circuit

Owns storage, entity creation, component instantiation, net membership, and private structural checks.

CircuitView

Read-only facade over one circuit. Readers and validators consume this instead of the mutable type.

Mutation Modules

CircuitHierarchy, CircuitElectrical, and CircuitDesignIntent provide cohesive write APIs over kernel-owned state.

Invariant Boundary

Circuit remains the only object with direct table ownership. Extracted mutation modules are thin kernel-owned facades that call privileged Circuit mutation hooks. They must not own alternate EDA meaning, cache connectivity, or mutate entity internals without the same checks that existed before this refactor.

Reader Migration

  1. Validation accepts CircuitView and uses deterministic rule suites.
  2. Logical writer reads from CircuitView; reader keeps mutating a Circuit through mutation modules.
  3. Schematic and PCB projections store or expose CircuitView for logical reads while retaining their own projection state.
  4. KiCad and future adapters consume projection objects and the view they expose; adapters never become owners of logical connectivity.
  5. Authoring and Python binding internals may change, but Python syntax and test behavior remain stable.

Validation Rules

Validation stays simple: named rule functions append to DiagnosticReport. Rule suites are deterministic arrays of functions for connectivity, electrical rules, default circuit validation, and PCB readiness. No dynamic plugin registry is introduced in this refactor.

Projection And Adapter Contract

  • Projection layers may reference logical components, pins, nets, ports, and module instances.
  • Projection layers own presentation, layout, board geometry, placement, routing, and readability/DRC diagnostics.
  • Projection layers never create, merge, split, or reconnect logical nets.
  • Adapters translate Volt-owned models to or from external formats and must remain outside core dependency direction.
