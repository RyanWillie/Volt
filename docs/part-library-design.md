# Part Library Design

Status: design proposal for review. APIs, file layouts, and JSON examples are
illustrative.

## Core rule

A library is to parts what a project is to a design: authored as declarative code, built
deterministically, validated by kernel-owned contracts, and evidenced by hashed,
kernel-loadable artifacts.

```text
PartDefinition    declarative core: identity, pin map, typed ratings, provenance
Symbol            presentation projection over the pin map; must line up
Footprint         physical projection over the pin map; must line up
OrderablePart     manufacturer, MPN, package; stable, library-owned
Sourcing          distributor, price, stock; volatile, design-owned snapshot
BOM               deterministic kernel projection over selected parts
```

A part has exactly one declarative source of truth. The schematic symbol and the physical
footprint are projections constrained by and validated against the pin map, never second
owners of pin meaning. Sourcing data (prices, stock, distributor SKUs) never lives in
library code.

## Problem

Volt's authoring model assumes that AI agents will write most board code, including
converting datasheets into parts. That changes the part library economics: authoring
verbosity is nearly free, but verification becomes the central cost. A footprint with a
transposed pad coordinate is syntactically perfect, passes review, and is fatal at
fabrication.

Today the codebase cannot support a trusted part pipeline:

- Parts exist only as Python objects. The C++ kernel cannot load, validate, hash, or diff
  a part independently of executing library code. This violates the rule that EDA meaning
  must not become Python-only.
- Two competing definition patterns exist: `Library.component(...)` binds pins, physical
  part, and symbol declaratively, while examples also assemble parts ad hoc with
  `select_part(...)` at instantiation. AI authors will generate both, inconsistently.
- Pin lists are duplicated between component definitions and symbol helpers, with no
  mechanical check that they agree. Drift is invisible in review.
- There is no library lifecycle: no independent build, no per-part artifacts, no content
  hashing, no version pinning from designs, no integrity check when a library changes.
- There is no BOM. Components carry manufacturer and MPN fields, but nothing aggregates
  selected parts into an orderable output, and there is no stored intent for DNP,
  per-instance selection overrides, or approved alternates.

## First-slice non-goals

- No package registry, index, or download tooling. Distribution is an ordinary Python
  package; pip/uv already solve it. Volt adds integrity, not distribution.
- No live distributor API integration in the kernel. Sourcing snapshots are design-side
  data produced by tooling above the kernel.
- No automatic symbol or footprint generation from the pin map. Projections are authored;
  the kernel validates lineup, it does not derive geometry or symbol aesthetics.
- No footprint geometry import from KiCad libraries as a source of truth. External
  libraries may serve as independent test oracles in library CI, never as the definition.
- No CAD execution at design build time. build123d/OCCT runs only in library development;
  designs consume frozen, hashed 3D artifacts.
- No design-time mutation of library parts. Deviations are explicit, manifest-visible
  acts, never in-place edits.

## Minimal vocabulary for the first slice

| Concept | Purpose | Owns / references |
| --- | --- | --- |
| `PartDefinition` | Declarative core of a part. | Owns identity (library namespace, name, version), the pin map, typed electrical ratings, provenance, and references to its projections. |
| Pin map | Single source of pin truth. | Pin name, number, role, typed semantics. Repeated names with distinct numbers (for example four `VDD` pins) are explicit. |
| `OrderablePart` | The stable physical identity. | Manufacturer, MPN, package name, footprint reference, pin-to-pad map, approved alternate MPNs. Extends today's `PhysicalPartSpec`. |
| Symbol projection | Presentation over the pin map. | References pins by name only. May not introduce, omit, or duplicate pins. |
| Footprint projection | Physical landing pattern. | Pads, optional courtyard and body polygons, plus a pin-to-pad map covering the pin map. Unmapped pads must declare a mechanical or thermal role. |
| Package generator | Reviewed parametric footprint producer. | IPC-7351-style families (SOIC, SOT, QFP, QFN, chip passives, headers) parameterized by datasheet-stated dimensions. |
| `PartModel3D` | Frozen 3D artifact reference. | File path, content hash, placement transform. Generated by library development tooling (for example build123d), committed as bytes. |
| `Library` | Versioned namespace of parts with a build. | Namespace, semantic version, part definitions, library-level provenance. |
| `.voltlib` bundle | Deterministic library build output. | Per-part kernel JSON, per-part SVG render, 3D artifacts, manifest with a content hash per part. |
| Design lock entries | Integrity pinning in the design manifest. | (library namespace, library version, part name, content hash) for every part a design consumes. |
| Sourcing snapshot | Volatile procurement data, pinned. | Design-owned file mapping MPN to distributor offers at a recorded time. Refreshing it is an explicit, diffable action. |
| BOM projection | Deterministic orderable output. | Walks component instances, resolves selected parts, groups by MPN, aggregates references and quantities, merges the sourcing snapshot by MPN. |

Deferred concepts: registry tooling, automatic projection generation, footprint import,
parametric part families beyond footprint generators, and lifecycle/obsolescence tracking
should be separate follow-up designs.

## Recommended first-slice decisions

### The kernel owns the part artifact

The keystone decision. A part serializes to a canonical, deterministic kernel JSON
payload that the C++ kernel can load, validate, and round-trip without executing Python.
Python remains the authoring surface; the artifact is the meaning. Content hashes, lock
entries, trust tiers, and BOM identity all attach to this byte representation, so it must
exist before any of them.

Location decision: part artifacts are sibling documents referenced by content hash, not
embedded payloads inside logical circuit JSON. A logical circuit may later record which
part hashes its instances consumed, and `.voltlib` manifests may index the same hashes,
but the canonical `volt.part` bytes stay independently loadable and hash-addressed. This
keeps lock entries, future manifests, and design-local snapshots attached to one byte
representation instead of duplicating embedded copies across designs.

The current implemented artifact format is `format: "volt.part"`, `version: 3`. Version 3 adds optional footprint-local `courtyard` and `body` polygons to the footprint projection; omitting either field means no such geometry was declared, not an empty extent.

### Projections must line up, and the kernel checks it

Symbol and footprint are validated against the pin map. Following the kernel rule that
structural integrity belongs at mutation and load boundaries while design quality belongs
in diagnostics, the lineup rules split into two fixed categories that do not depend on
which path exercised them:

Structural — rejected both when a part is assembled and when an artifact is loaded:

- no symbol pin outside the pin map, and every pin map entry appears in the symbol
  exactly once;
- every pin-to-pad map entry references a real pin and a real pad; multi-pad pins (for
  example a tab plus a leg) are explicit in the pin-to-pad map;
- repeated pin names resolve to distinct pin numbers.

Design-quality — reported as diagnostics over assembled and loaded parts:

- a pin with no mapped pad;
- a pad neither mapped to a pin nor declaring a mechanical/thermal role;
- pad geometry consistency: undeclared overlaps, inconsistent pitch within a row.

These are cheap checks in the existing ERC style. They are the mechanical check that
catches AI-authored drift that human review reliably misses.

### Shrink the surface AI is trusted with

Pin tables are text and transcribe reliably. Package drawings are where errors hide.
Footprint authoring should therefore route through reviewed package generators wherever a
standard package exists: the AI supplies a handful of datasheet-stated dimensions (pitch,
span, pad size, pin count) instead of dozens of raw coordinates. Generators are reviewed
once, covered by golden tests, and trusted thereafter. Hand-placed pad coordinates remain
the escape hatch for irregular parts, and that escape hatch is exactly where the strictest
verification (render review plus independent oracle diff) concentrates.

### Provenance and verification state are first-class data

A trusted part is a recorded, inspectable status, not a folklore property:

```text
provenance:
  datasheet     = document identifier and revision
  authored_by   = human | agent identifier
  derived_from  = generator name and parameters, if any

verification:
  generated           artifact exists, lineup contracts pass
  cross_checked       geometry diffed against an independent oracle
  render_reviewed     per-part SVG approved by a human
  hardware_validated  part fabricated and verified on a board
```

Diffs need independent references. A test written by the same agent from the same
datasheet misreading agrees with the wrong code. Acceptable oracles include a second
independent derivation or an identically-named external library footprint used strictly
as a comparison witness in library CI.

### One build pipeline, two entry points

`library.build()` runs standalone in the library repository and emits the `.voltlib`
bundle. When a design's project includes a library, the project build invokes the same
build code path, folds the library's lineup diagnostics into the project diagnostic
report with usage context ("pin/pad mismatch in `acme.parts:AP1117-15`, used by U3"), and
records part hashes in the design manifest. Because part artifact generation is
deterministic, building from source and consuming a prebuilt bundle produce identical
bytes; hash equality is the bridge between the two modes, and rebuilding-then-comparing
against a published bundle is a free determinism check in CI.

Exception: 3D artifacts are not rebuilt by consumers. CAD generation is a
library-developer operation; its committed output is verified by hash, never regenerated,
keeping heavyweight and platform-sensitive CAD dependencies out of design builds.

### Libraries are immutable from the design's side

A project build may build a library; it may not patch one. Design-specific deviations are
explicit kernel-visible acts — a design-local library, or a declared per-instance
override — that appear in the manifest and the BOM. A part consumed under a library's
name and version is byte-identical to what that library published, always.

### BOM is a kernel projection over three layers

1. Component definition — pins and semantics. Stable, library-owned. Exists today.
2. Orderable part — manufacturer, MPN, package, footprint, ratings, 3D. Stable,
   library-owned. Largely exists as `PhysicalPartSpec`.
3. Sourcing — distributor, SKU, price, stock. Volatile, design-owned, pinned in a
   sourcing snapshot. Never in library code, so price churn cannot invalidate part hashes.

The BOM writer walks component instances, resolves selected parts, groups by MPN,
aggregates reference designators and quantities, merges the sourcing snapshot by MPN, and
emits deterministic `bom.json` and `bom.csv` into the project bundle alongside schematic
and PCB outputs.

Three pieces of stored design intent land with the first slice, while the shape is cheap
to change:

- **DNP** — do-not-populate as a typed flag on the component instance, not a property
  string;
- **per-instance selection override** — this divider needs the 0.1% variant even though
  the library default is 1%;
- **approved alternates** — substitute MPNs on the orderable part, so assembly does not
  stall on single-source lines.

A `BOM readiness` validation entry point joins the existing general, connectivity, ERC,
and PCB-readiness entry points: every populated instance resolves to an orderable part,
DNP intent is explicit, alternates are structurally compatible.

This slice overlaps the assembly-handoff scope of VOL-188 (BOM and component placement
artifacts in the manufacturability epic). When this design is ticketed, VOL-188 should be
re-scoped to consume the BOM projection defined here rather than specifying a second one;
the placement (CPL) artifact remains VOL-188 scope.

## Authoring shape

Library authoring stays close to today's `Library.component(...)`, with the pin map as
the single pin source and projections referencing it. `PinSpec` role strings are Python
authoring sugar only: they lower into the kernel's canonical typed pin semantics
(`terminal_kind`, `direction`, and related fields), and the part artifact stores only the
typed form — the persisted format has no role field:

```python
LIB = volt.Library("acme.parts", version="1.2.0")

AP1117_15 = LIB.part(
    "AP1117-15",
    pins=[
        volt.PinSpec("GND", 1, role="ground"),
        volt.PinSpec("VO", 2, role="power_output"),
        volt.PinSpec("VI", 3, role="power_input", voltage_range=(2.5, 18.0)),
    ],
    orderable=volt.OrderablePart(
        manufacturer="Diodes Incorporated",
        part_number="AP1117E15G-13",
        package="SOT-223-3",
        footprint=volt.packages.sot223(pins=3, tab_pad="4"),
        pin_pads={1: "1", 2: ("2", "4"), 3: "3"},
        approved_alternate_mpns=("AP1117E15G-7",),
        model_3d=volt.PartModel3D("models/sot223.step"),  # frozen bytes, verified by hash
    ),
    symbol=regulator_symbol,          # references pins by name; lineup is validated
    extensions={"provenance": {"datasheet": "AP1117 rev 24"}},
    prefix="U",
)
```

Designs consume parts by ordinary import, and the project build pins what it consumed:

```python
from acme_parts import LIB

u3 = design.instantiate(LIB.AP1117_15, ref="U3")
design.dnp(c7)                          # stored intent, visible in BOM
u4 = design.instantiate(LIB.AP1117_15, ref="U4",
                        select=LIB.AP1117_15.alternate("AP1117E15G-7"))
```

```json
{
  "libraries": [
    {
      "namespace": "acme.parts",
      "version": "1.2.0",
      "parts": [
        {"name": "AP1117-15", "hash": "sha256:..."}
      ]
    }
  ]
}
```

## Sequencing

1. Kernel-owned part artifact: canonical serialization, loader, content hashing. The
   keystone; everything else attaches to it.
2. Lineup-contract diagnostics over assembled and loaded parts.
3. `library.build()` and the `.voltlib` bundle; per-part SVG render; design-manifest lock
   entries; in-project library builds through the same code path.
4. BOM projection with DNP, per-instance overrides, and alternates; BOM-readiness
   validation entry point.
5. Package generator library with golden tests; provenance and verification tiers;
   sourcing snapshot format.

Steps 1, 2, and 4 are kernel shape decisions and should land before depth investment
elsewhere. Steps 3 and 5 are layered conveniences that can iterate.

## Open questions

- How does a design-local library relate to project structure: a `Library` declared
  inline in project code, a sibling package, or both?
- What is the minimum dimensional-sanity rule set for footprints that is useful without
  guessing fabrication capabilities (which belong to DRC and net classes)?
- Where do part-level typed ratings stop and selected-part attributes begin when the same
  fact (for example voltage rating) exists today on the selected part?
- Does the verification tier belong in the part artifact itself or in the library
  manifest, given that hardware validation happens after a library version is published?
