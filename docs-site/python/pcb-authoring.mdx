title: PCB Authoring
description: Describe PCB layers, footprints, placements, routing, zones, keepouts, text, validation, and output.
---

PCB authoring projects the logical design into physical board data. Board-ready real parts should attach selected physical part data with a first-class `volt.Footprint` object; board placement then resolves that object-owned geometry automatically.

## Board setup

```python
board = design.board("First Board LED")

front = board.add_layer("F.Cu", role="copper", side="top")
back = board.add_layer("B.Cu", role="copper", side="bottom")

board.set_layer_stack((front, back), thickness=1.6)
board.set_rectangular_outline(origin=(0.0, 0.0), size=(32.0, 18.0))
```

## Footprints

```python
resistor_0603 = volt.Footprint(
    ("passives", "R_0603_1608Metric"),
    pads=(
        volt.FootprintPad.surface_mount(
            "1",
            at=(-0.75, 0.0),
            size=(0.80, 0.95),
            shape="rounded_rectangle",
        ),
        volt.FootprintPad.surface_mount(
            "2",
            at=(0.75, 0.0),
            size=(0.80, 0.95),
            shape="rounded_rectangle",
        ),
    ),
    courtyard=((-1.6, -0.9), (1.6, -0.9), (1.6, 0.9), (-1.6, 0.9)),
    body=((-0.8, -0.4), (0.8, -0.4), (0.8, 0.4), (-0.8, 0.4)),
)

r1.select_part(
    manufacturer="Yageo",
    part_number="RC0603FR-07330RL",
    package="0603",
    footprint=resistor_0603,
    pin_pads={1: "1", 2: "2"},
)
```

The footprint identity is still serialized as the selected part reference, but the `volt.Footprint` object carries the pad, courtyard, and body geometry needed for board placement. Omit `courtyard` or `body` when the footprint does not declare that polygon; an omitted polygon remains distinct from an invalid empty polygon. Polygon arrays use canonical boundary vertices and must not repeat the first vertex at the end. `board.cache_footprint(...)` remains available for importers, explicit board-owned snapshots, and low-level tests.

## Placement and routing

```python
with board.layout(unit=1.0, grid=0.5) as layout:
    resistor = layout.two_pad(r1).at((15.0, 7.0)).right()
    led = layout.place(d1, at=resistor.center.right(9.0), orient="left")

    layout.route(led_a, layer=front, width=0.25).at(resistor.end).right(2.0).toy(
        led.A
    ).to(led.A)

    drop = layout.node(led.K.left(2.0))
    layout.route(gnd, layer=front, width=0.30).at(led.K).to(drop)
    layout.via(gnd, at=drop, start_layer=front, end_layer=back)
```

Layout routes are hand-placed physical copper: `.at(...)`, `.tox(...)`, `.toy(...)`,
directional jogs, and `.to(...)` lower to kernel-owned tracks. `layout.via(...)` lowers
to the same via model as `board.add_via(...)`. Tracks and vias reference existing logical
nets. Board data implements connectivity; it does not define the netlist.

When `grid` is set, explicit authoring coordinates snap to that board grid while
footprint pad anchors stay exact. `.to(...)` routes default to octilinear geometry, so
arbitrary endpoints become 0/45/90 degree segments. Use `mode="direct"` only when a
non-octilinear track is deliberate.

`board.add_track(...)` remains available when an importer or advanced authoring workflow
needs to provide exact point lists deliberately.

```python
# Low-level facade calls remain available for exact imported board geometry.
board.place(r1, at=(15.0, 7.0), rotation=0.0, side="top")
board.add_track(
    led_a,
    layer=front,
    points=[(15.75, 7.0), (20.0, 3.0), (24.75, 7.0)],
    width=0.25,
)
```

The same route can be authored from pad and local anchors instead of a raw point list:

```python
with board.layout(grid=0.5) as layout:
    layout.connect(
        resistor.end,
        led.A,
        layer=front,
        through=(resistor.end.right(2.0),),
        width=layout.rule("min_track_width"),
    )
```

Use `layout.bundle(...)` when several explicit anchor pairs should be authored together.
Each bundle item is a `(start, end)` or `(start, end, through)` tuple and lowers through
the same `layout.connect(...)` path. Batch helpers preserve the layout cursor.

## Zones, keepouts, and text

```python
with board.layout(grid=0.5) as layout:
    layout.zone(
        net=gnd,
        layers=(front, back),
        at=board.corner("top-left").right(2.0).down(2.0),
        size=(28.0, 14.0),
    )

    layout.keepout(
        layers=(front,),
        at=(4.0, 4.0),
        size=(4.0, 4.0),
        restrictions=("copper", "via"),
    )

    layout.text("Volt", at=board.edge("bottom").center().up(1.0), layer=front, size=1.2)
```

`layout.rect(...)` and `layout.polygon(...)` return reusable anchor outlines for
zones, keepouts, cutouts, or board-local construction. These helpers compose geometry
only; zones, keepouts, text, tracks, and vias still lower through the kernel-owned board
mutation APIs and serialize as ordinary board data.

## Repeated layout

```python
with board.layout(grid=0.5) as layout:
    row = layout.distribute(count=4, start=board.edge("left").center(), end=board.edge("right").center())
    aligned = layout.align(row, axis="y", target=board.center)
    mirrored = layout.mirror(aligned, axis="x", about=board.center)

    layout.fanout(
        aligned,
        layer=front,
        direction="up",
        distance=1.0,
        net=gnd,
        via_layers=(front, back),
    )
    layout.stitch(gnd, at=mirrored, start_layer=front, end_layer=back)
```

Placement composition returns anchors and result records; board meaning still appears only
when the layout lowers to `place`, `connect`, `via`, `zone`, `keepout`, or `text`.
Batch helpers preserve the cursor so repeated layout does not depend on iteration order.

## Validate and write

```python
readiness = design.validate_for_pcb()
board_report = board.validate()

if readiness.has_errors or board_report.has_errors:
    for report in (readiness, board_report):
        for diagnostic in report:
            print(diagnostic.severity, diagnostic.code, diagnostic.message)

board.write_json("board.volt.pcb.json")
board.write_svg("board.svg")
export = board.write_kicad_pcb("board.kicad_pcb")

for warning in export.warnings:
    print(warning.fabrication_impact, warning.construct, warning.message)
```

`board.validate()` catches board-level issues such as missing selected parts, unresolved footprint geometry, invalid layer usage, and DRC-style projection problems.
Project runs convert fab-critical KiCad export losses into `PCB_KICAD_FAB_EXPORT_LOSS`
diagnostics under `pcb.kicad_export`, so expected-diagnostics policy can gate known
handoff gaps. The diagnostic `rule` is the KiCad loss construct, such as
`board.zone`, so expectations can allow one known omission without masking another.
Informational adapter losses remain visible in `export.warnings`.

For local KiCad verification without making CI depend on KiCad, run:

```bash
python3 scripts/verify-kicad-fab-export.py board.kicad_pcb --output-dir artifacts/kicad-fab --drc
```

The script exports Gerbers and drill files with `kicad-cli` when it is installed, and
skips cleanly when `kicad-cli` is absent.
