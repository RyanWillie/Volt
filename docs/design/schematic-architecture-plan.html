Volt schematic plan

Schematic is a projection, not a second circuit.

The goal is a schematic system that can be authored from Python, rendered to SVG, and eventually imported/exported through KiCad, while preserving Volt's core rule: logical connectivity remains owned by the C++ circuit kernel.

North star
Logical circuit owns meaning. Schematic owns presentation. Libraries provide reusable definitions. Import/export adapters translate at the edges. Python is syntax over kernel-owned schematic data.

This means drawing a wire must never silently create a second netlist. A schematic wire can visualize an existing NetId. A Python helper may feel like Schemdraw, but any operation that changes connectivity must first call the logical circuit mutation API, then create projection data over the result.

Layer split

Logical circuit

Owns component definitions, component instances, pins, nets, selected parts, modules, port bindings, and typed electrical semantics.

Schematic projection

Owns sheets, symbol placement, symbol pin presentation, wire geometry, labels, no-connect markers, and annotations referencing logical entities.

Adapters

KiCad, SVG, PDF, and future tool formats import/export to Volt's canonical model. They are not the internal schema.

Architecture
Library Component definitions, symbol definitions, and physical part templates with provenance.
Logical Concrete components, pins, nets, modules, and selected physical parts.
Schematic Sheets, placed symbols, labels, wire runs, and drawing metadata.
Renderer SVG first, then PDF/PNG if needed. Deterministic and inspectable.
Adapters KiCad symbol/schematic import/export once the internal model is stable.

Key decisions
Decision 1

Use structured vector primitives as the canonical symbol format.

Do not store raw SVG as the source of truth. Store lines, rectangles, circles, arcs, text, and pin anchors in the kernel. SVG is a renderer output.

Decision 2

Keep KiCad as an adapter, not the internal model.

KiCad import/export is valuable, but Volt's schematic model should be small, deterministic, and kernel-owned. KiCad libraries translate into Volt component, symbol, and physical part structures.

Decision 3

Let Python feel like Schemdraw without becoming the model.

Python can expose fluent placement and drawing helpers, but every meaningful operation must lower into C++ schematic projection data or logical circuit mutations.

Decision 4

Defer auto-layout until explicit placement works.

Auto-layout is tempting and expensive. Start with deterministic explicit placement, then add helper defaults, then consider layout engines after the renderer proves the model.

Library model

Canonical split

  • ComponentDefinition: logical pins and electrical semantics.
  • SymbolDefinition: schematic drawing and pin presentation.
  • PhysicalPart: manufacturer, package, footprint, pin-pad mapping.
  • LibrarySource: provenance, such as KiCad symbol library import.

Import/export direction

  • Import KiCad symbols into Volt symbol definitions.
  • Import KiCad footprint refs into selected physical part data.
  • Export Volt schematics to SVG first, KiCad later.
  • Do not require KiCad libraries for the first working schematic.
Python shape

The first Python API should be compact and explicit. It should feel natural for code-authored schematics while making the ownership boundary obvious. 

d = volt.Design("front_end")

r1 = d.R(resistance=10_000, ref="R1")
c1 = d.C(capacitance=100e-9, ref="C1")
vin = d.net("VIN", kind="power", voltage=3.3)
out = d.net("OUT")

vin += r1[1]
out += r1[2], c1[1]

sch = d.schematic("main")

r1s = sch.place(r1, at=(40, 20), symbol="resistor_eu")
c1s = sch.place(c1, at=(70, 20), rotate=90)

sch.wire(vin, path=[(10, 20), r1s.pin(1)])
sch.wire(out, path=[r1s.pin(2), c1s.pin(1)])
sch.label(vin, "VIN", at=(10, 20))
sch.render_svg("front_end.svg")
If sch.wire() is asked to connect two pins that are not already on the same logical net, it must either reject the operation or explicitly perform the logical circuit connection first. Silent schematic-only connectivity is not allowed.
Work packages
PR 1

Schematic vocabulary design

Lock names, ownership, references, coordinate units, symbol primitive set, and serialization boundaries in a design document before implementation.

PR 2

Kernel IDs and schematic primitives

Add typed IDs, geometry/value types, symbol vector primitives, and basic storage tables. No renderer yet.

PR 3

Symbol definition model

Add symbol definitions, symbol pins, pin anchors, and fixture tests for common two-pin symbols.

PR 4

Sheet and symbol instance model

Place symbol instances on sheets and reference logical component instances. Add consistency checks for missing or wrong references.

PR 5

Schematic JSON

Persist and round-trip schematic projection data separately from logical circuit JSON or as an explicitly named projection section.

PR 6

Python placement API

Add d.schematic(), sch.place(), symbol lookup, and read-only handles over kernel schematic objects.

PR 7

SVG renderer

Render placed symbols, text, simple wires, labels, and basic page geometry to deterministic SVG.

PR 8

Wire and label projection

Add wire runs and labels referencing existing NetIds. Validate that visual connectivity does not contradict the logical circuit.

PR 9

KiCad adapter design

Map KiCad symbols and schematics into Volt structures after the canonical model is proven by SVG output and Python authoring.

Out of scope now

No full KiCad clone

KiCad compatibility is an adapter track. The first schematic model should not inherit KiCad's full connection-graph complexity.

No automatic layout

Explicit placement first. Automatic layout can come later when the model and renderer are stable.

No ERC expansion

Schematic readiness diagnostics are allowed. Broader electrical-rule work stays in the validation track.

Acceptance
Milestone Evidence
Kernel model exists Tests can create sheets, symbol definitions, and symbol instances referencing logical components.
Projection does not own connectivity Wire runs reference existing NetIds and cannot create hidden pin-to-net membership.
Python is ergonomic A small circuit can place symbols and wires with concise Python syntax.
Renderer works A deterministic SVG shows placed components, visible pins, labels, and wire runs.
Adapters are cleanly separated KiCad code depends on Volt schematic structures, not the other way around.
This page intentionally plans the schematic foundation before Python drawing sugar or KiCad interoperability. The goal is a useful schematic layer that remains faithful to Volt's kernel-owned EDA semantics.
