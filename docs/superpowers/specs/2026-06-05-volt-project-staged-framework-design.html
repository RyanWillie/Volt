VOL-170 design

Staged Project Framework

A Volt project should feel like a guided electronics workflow: design the netlist, project the schematic, project the PCB, run checks, and write a deterministic product result.

Date: 2026-06-05

Scope: Python framework foundation over existing kernel-owned models.

Decision: Make staged decorators and stage-attached tests the product-workflow path. Stages compose the actual product layer; they must not wrap a parallel build_* framework.

Why This Exists

Volt already has meaningful layers: logical design, schematic projection, PCB projection, validators, and deterministic serializers. Loose Python scripts can produce those models, but the source has no canonical entrypoint and the outputs have no stable project-level shape.

Framework value

The framework gives users, tests, agents, CI, and the future app named checkpoints they can run, inspect, validate, and write without guessing how helper functions are organized.

Run

Run the whole product or stop after a stage such as schematic.

Diagnose

Group findings by stage and model so feedback says what failed and where.

Iterate

Edit Python, rerun a named stage, compare deterministic artifacts, repeat.

Public Shape

import volt

project = volt.Project(
    "status-led",
    version="0.1.0",
    description="Single-board status LED module",
)

@project.design
def design():
    d = volt.Design("status-led")
    # Compose the netlist here: components, nets, selected parts, and logical intent.
    return d

@project.schematic
def schematic(design):
    sheet = design.schematic("Main")
    # Compose the schematic here: symbols, existing-net wires, labels, and ports.
    return sheet

@project.board
def board(design):
    pcb = design.board("Main")
    # Compose the PCB here: outline, placement, routing, copper, and features.
    return pcb

@project.design.test
def test_status_led_netlist(check):
    check.net("VCC").connects("J1.1", "R1.1")
    check.net("GND").connects("J1.2", "D1.K")

result = project.run()
result.write("dist/status-led.volt")

Small projects can stop after @project.design. Larger products can add stages as the workflow becomes real. Returning known Volt models registers them in deterministic stage order. Product-specific tests live next to the stage whose intent they protect.

Composition belongs in the stage

If a decorated stage only calls a top-level build_design, build_schematic, or build_board wrapper, the framework has added ceremony. Helper functions are still useful for repeated local blocks, but the decorated stage is the product composition point.

Stages

Design

Canonical netlist

Creates the primary Design root for VOL-170. Connectivity and EDA meaning stay kernel-owned.

Schematic

Presentation

Projects existing logical connectivity into sheets, symbols, wires, labels, and ports.

Board

Physical PCB

Projects the logical design into board outline, stackup, placements, routes, copper, and features.

Result

Product snapshot

Collects models, diagnostics, artifacts, metadata, and pass/fail status for one run.

Stage Function signature Accepted return values in VOL-170 Checks
@project.design fn() Design or iterable of Design design.validate() plus @project.design.test hooks
@project.schematic fn(design) Schematic or iterable of Schematic schematic.validate(), schematic.validate_readability(), and @project.schematic.test hooks
@project.board fn(design) Board or iterable of Board design.validate_for_pcb(), board.validate(), and @project.board.test hooks
@project.design.test fn(check) Product-intent assertions over the stage result Failures become project test diagnostics

Run Model

project.run()                                # design -> schematic -> board
project.run_through(project.design)          # netlist diagnostics and design tests
project.run_through(project.schematic)       # design plus schematic checks

result.stage(project.design).ok
result.diagnostics.errors(stage=project.board)
result.model(project.design, "status-led")
result.board("Main")

Partial runs are the practical reason for stages. A user can stop after the layer they are actively editing and get immediate diagnostics and product tests without running later work. Stage handles avoid stringly-typed framework APIs.

One canonical path first

VOL-170 should make one design flowing into one schematic and one board feel excellent, while allowing stages to return multiple known models in deterministic order. Large products should split repeated local blocks into helpers, not split every subcircuit into a project stage.

Stage Tests

Project tests are first-class product-intent checks. They are different from kernel diagnostics: diagnostics report general EDA correctness, while project tests lock down what this product must keep true.

@project.design.test
def test_usb_lines(check):
    check.net("USB_DP").connects("J_USB.D+", "U1.PA12")
    check.net("USB_DM").connects("J_USB.D-", "U1.PA11")
    check.no_connection("USB_DP", "USB_DM")
    check.no_connection("+12V", "+3V3")

@project.schematic.test
def test_schematic_review_surface(check):
    check.schematic("STM32 USB Buck").has_component("U1")
    check.schematic("STM32 USB Buck").has_net_label("+3V3")
    check.schematic("STM32 USB Buck").has_net_label("GND")

@project.board.test
def test_board_readiness(check):
    check.board("STM32 USB Buck").has_all_components_placed()
    check.board("STM32 USB Buck").has_all_footprints_resolved()
Layer Useful tests Benefit
Netlist Rails reach expected pins; connector signals are not swapped; important nets are isolated; required parts/footprints are selected. Fast source-of-truth regression checks before schematic or PCB work can obscure the problem.
Schematic Critical components are present; important labels and ports exist; review-critical regions are populated. Protects review usability without making placement itself the source of connectivity.
PCB Board outline exists; all components are placed; all footprints resolve; critical connector placement remains stable. Turns layout readiness into a repeatable project gate.
Result All stage tests and default diagnostics pass for the requested run depth. Gives CI, agents, and the app one integrated pass/fail surface.

Testing is a framework feature

The normal pytest wrapper can be tiny: assert project.verify().ok. Detailed failures live in ProjectResult with stage, test name, severity, and message.

Boundary

Project tests assert product intent over kernel-owned models. They do not mutate nets, own schematic placement, route boards, or become hidden EDA semantics.

Diagnostics

Diagnostics should be first-class result data, not a side effect of writing JSON. Each collected diagnostic gets stable source metadata while preserving the kernel diagnostic fields.

{
  "source": "pcb:Main",
  "stage": "board",
  "report": "pcb.board",
  "severity": "warning",
  "code": "PCB_UNROUTED_NET",
  "message": "...",
  "entities": [{"kind": "net", "index": 2}]
}
Question Result API
Did the run fail? result.ok is false when any collected diagnostic is an error.
What layer failed? result.stage(project.board).diagnostics or result.diagnostics.errors(stage=project.board).
What should the app show? The manifest points from diagnostics to stable model and artifact paths.

The bundle is the future app and agent contract. It omits timestamps and machine-local paths. Artifact ordering comes from stage order and model registration order. JSON helper output uses sorted keys where the project framework owns the document.

Manifest minimum

The manifest includes format, schema version, project metadata, run parameters, stages run, generated artifact records, diagnostics summary, test summary, and stable relative paths.

Out Of Scope

  • No package manager, dependency resolver, app runtime, database, history system, or persistent cache.
  • No Python-only EDA checks that alter the meaning of designs, schematics, boards, rules, or constraints.
  • No component 3D model asset bundle in VOL-170; the bundle layout leaves room for assets later.
  • No full multi-board assembly model in VOL-170; use stable registries so the later assembly work is additive.
  • No new C++ kernel mutation APIs unless an existing serializer or validator is missing.
