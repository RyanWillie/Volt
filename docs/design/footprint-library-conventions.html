Footprint Library Conventions

Volt footprint libraries are normalized geometry data consumed by PCB projection. They do not select parts, merge nets, or patch circuit intent.

Boundary: selected physical parts own package identity, a FootprintRef, and logical pin-to-pad mappings. Board-ready Python parts should carry a first-class volt.Footprint object with the corresponding geometry, while board projections place existing components and resolve geometry without choosing library parts.

Structural Data

A footprint definition is a library-qualified name plus ordered pads. Empty names, duplicate footprint references, duplicate pad labels, non-finite coordinates, non-positive sizes, invalid layer sets, and malformed through-hole drill data are rejected at the kernel or load boundary.

Design Diagnostics

Missing selected parts, unresolved footprint references, missing electrical pad mappings, and accidental mappings to mechanical pads are design-quality problems reported through diagnostics and validation.

Geometry

  • All footprint coordinates, pad sizes, and drill diameters are millimeters.
  • Pad coordinates are footprint-local centers. Placement transforms convert them to board space.
  • Pad IDs are implicit and stable by definition order: footprint_pad:0, footprint_pad:1, and so on.
  • Pad labels are the selected-part mapping keys. They must be unique inside one footprint.
  • A selected logical pin may map to multiple unique electrical pad labels when the physical package exposes tied lands, such as a regulator tab.
  • SMD pads use front_smd or back_smd layer sets. Through-hole electrical pads use through_hole.
  • Non-plated holes use mechanical_hole layers and must declare a mechanical role such as mounting or mechanical_support.
  • Mechanical pads do not require selected-part pin mappings. Mapping a logical pin to one is diagnosed as invalid design intent.

Current Built-In Coverage

The built-in library covers first-board examples: common 0402/0603/0805 passives, 0603 LEDs and diodes, SOD-123 diodes, SOT-23, SOT-23-6, and SOT-223 tabbed packages, SOIC and TSSOP packages, 1x02 and 1x04 headers, 2x05 programming headers, a simple terminal block, a micro USB-B receptacle, and the LQFP-64_10x10mm_P0.5mm MCU package used by the STM32 benchmark library.

Growth Path

Future footprint growth should lower into the same Volt-owned FootprintDefinition values before board use. A package label or footprint reference alone is provenance, not board-ready geometry. A safe data-driven path is a small checked-in footprint-definition format that validates into the kernel types. A safe KiCad path is a narrow adapter that imports only supported pad geometry, layer sets, drill data, and mechanical roles, records unsupported constructs as loss diagnostics, and never treats a KiCad footprint string as a selected physical part by itself.
