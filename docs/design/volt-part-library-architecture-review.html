This page refines the part-library seam. It does not replace the post-Circuit architecture review or amend an accepted ADR.
01 · Decision

A library is a buildable product, not a folder of helpers.

Adopt a first-class, C++-owned part-library system that mirrors the agreed authoring, service and artifact planes. Python remains the pleasant way to write and import libraries; the compiled bundle is the portable meaning.

Gold target: exact parts are complete, typed and independently reopenable.

A publishable PartDefinition binds one reusable logical contract to one primary manufacturer part number, exact typed characteristics, pin constraints, symbol, footprint and required assets. A project persists the chosen part key and content digest - never a floating search query.

01 / DEFINE

Declarative source

Python or C++ declarations, generators, assets and ordinary unit tests.

02 / COMPILE

Native services

C++ validates, hashes, queries, resolves and emits the library bundle.

03 / CONSUME

Immutable closure

Projects snapshot only the selected parts and assets required by their builds.

Electrical meaning is explicit. Typed characteristics and limits are mandatory kernel data when a contract requires them. Optional simulator models attach to that meaning; they do not replace it.
Companion relationship. The broader post-Circuit review remains the owner of the three-plane project architecture, multiple Boards, CompiledBoard, typed artifact graph, selected exports and C++-first bundle API. This review supplies the missing reusable-part boundary.

03 · Vocabulary

Separate the things users currently call “a part”.

Clear names prevent generic electrical intent, exact procurement identity, physical geometry and reusable circuits from collapsing into one object.

01

ComponentDefinition

volt.components.resistor

The reusable logical contract: pins, pin semantics, required instance inputs and compatible part shape. It may be useful for schematic authoring while still unresolved for manufacturing.

02

PartDefinition

yageo/RC0402FR-0710KL

One exact, primary manufacturer + MPN identity with intrinsic characteristics, ratings, recommended ranges, absolute limits, package binding, pin-to-pad mapping and provenance.

03

FootprintDefinition

ipc/chip-0402-n

A reusable physical landing pattern with pads, fabrication, assembly, body, courtyard and markings. One qualified identity resolves to one geometry.

04

Model3DAsset

sha256:… .glb / .step

A content-addressed footprint-relative model. GLB serves Vault and BoardScene; STEP is optional mechanical interchange. Consumers verify bytes and never rerun CAD generation.

05

SimulationModelDefinition

optional · hash-addressed

A separate model contract and backend payload reference with terminal bindings, parameters, fidelity and supported analyses. Ratings remain useful without it.

06

PartLibraryBundle

ryan/parts @ 1.0.0 + digest

The immutable published catalogue: manifest, canonical definitions and required content-addressed assets. It opens through C++ without executing authoring Python.

07

ModuleDefinition

esp32_minimal_power

A reusable multi-component circuit or reference design. It can consume exact parts, carry tests and later carry simulation cases, but it is not an atomic part.

Manufacturing readiness is visible. An unresolved ComponentDefinition is allowed during logical exploration. It must fail Board/BOM readiness until a compatible exact PartDefinition is pinned. A manufacturable PartDefinition never means “some resistor in this family”.

05 · Electrical contract

Critical electrical facts are normal part data.

This is a hard architecture decision, not an implied convention. Any fact needed for selection, validation, ERC or future simulation is typed, kernel-owned, serializable and inspectable after Python is gone.

Authoring → native build → canonical bundle → native reopen → selected instance must preserve electrical meaning exactly.

Unknown values stay unknown. The library produces completeness diagnostics; it never guesses from a category string, net name, filename or an agent’s likely interpretation of a datasheet.

Always kernel data

Typed values, tolerances, ranges, pin semantics, exact MPN, package/pad mapping, provenance and required contracts.

Optional attachment

Vendor or approximate simulation models, 3D interchange, measured evidence and analysis-specific parameter sets.

Never inferred truth

Values guessed from names, category strings, convenient defaults, distributor listings or solved simulation outputs.

Use capability contracts, not one universal required-field list

A resistor contract can require nominal resistance, tolerance, rated power and maximum working voltage. A regulator contract can require supply/output ranges, output-current capability and correctly typed power pins. An MCU contract can require supply-domain and I/O limits without pretending every datasheet curve is a scalar. Each contract is versioned kernel vocabulary.

// Illustrative target. Names are not accepted API.

    resistance:              Quantity<Resistance>,

First migration priority: close today’s fidelity gap before adding a bundle. The native v4 model supports part-level attributes, but Python constructs an empty native map; four of ten native dimensions also lack generic Python parsing. Packaging incomplete meaning would merely make the incompleteness portable.
06 · Simulation boundary

Simulation-ready, not simulation-specific.

Typed ratings are not a simulator model. The catalogue should preserve both without coupling Volt’s canonical meaning to SPICE, IBIS, an internal solver or a Python callback.

Canonical designPart facts + Circuit connectivity
→
Optional projectionSimulationModelDefinition + hashed backend payloads
→
Analysis artifactSimulationGraph + SimulationResult

Model attachment owns

  • model kind and supported analyses;
  • terminal schema and part-pin bindings;
  • typed parameter bindings and corners;
  • fidelity: ideal, approximate, vendor or measured;
  • limitations, provenance, licence and content digest;
  • backend payload refs such as SPICE, IBIS or S-parameter files.

Readiness diagnostics own

  • no model for the requested analysis;
  • model terminals or units do not line up;
  • required part attributes are missing;
  • backend cannot represent the chosen model;
  • model fidelity or ignored effects are insufficient;
  • PCB parasitics are absent when a case expects them.
volt.part
├── typed intrinsic attributes
├── typed pins and constraints
├── symbol / footprint / 3D references
└── optional SimulationModelRef
      ├── model artifact digest
      ├── part-pin → model-terminal bindings
      └── model-parameter → typed-attribute bindings
             │
             ▼
      volt.simulation_model
      ├── terminal and parameter schemas
      ├── supported analyses and fidelity
      ├── provenance, limits and licence
      └── hashed backend payload references
Scope line: this review reserves the correct attachment boundary; it does not implement a solver, SPICE integration or simulation API. Absence of a model cannot invalidate an otherwise useful schematic/PCB/BOM part. It only affects validate_for_simulation(requested_analysis).

Simulation reduces uncertainty; it does not prove a physical product. Hardware validation and manufacturing evidence remain the loop-closing proof, and their attestations must state exactly what was tested.

08 · Library workflow

A personal catalogue behaves like a serious software package.

The source repository is pleasant to edit and test. The published product is language-neutral. Local and Git dependencies solve the first useful workflow; a registry is discovery infrastructure for later.

Personal and team libraries

Use local/editable path or pinned Git dependencies. The project lock records the exact built digest. A thin package can preserve from ryan_parts import preferred while loading native LibraryPartRef handles.

Official catalogue

Live in a separate, curated volt-standard-parts repository with its own release cadence, licensing, review evidence and immutable bundles. Start narrow and excellent.

Published consumers do not execute library authoring source. Editable development may rebuild it first, but both paths must produce byte-identical canonical entries. A future registry indexes signed bundles; it never becomes an alternate owner of part meaning.

09 · Artifact closure

Ship what runtime needs. Keep evidence beside it.

A proper library artifact may contain canonical JSON internally, but users consume one versioned bundle with declared media types and hashes - not a loose .volt.json development file or an opaque Python object graph.

Logical bundle contents

ryan-parts.voltlib ├── manifest # namespace, version, compatibility, licence, digest index ├── definitions/ │ ├── components/<digest> # reusable logical contracts │ └── parts/<digest> # exact canonical PartDefinition artifacts ├── projections/ │ ├── symbols/<digest> │ └── footprints/<digest> ├── assets/ │ ├── models3d/<digest>.glb # viewer-ready bytes │ ├── models3d/<digest>.step # optional mechanical interchange │ └── simulation/<digest> # optional backend payloads └── attestations # compact evidence refs keyed to immutable digests

Runtime bundle includes

  • canonical definitions and schemas;
  • every referenced projection and actual asset byte;
  • licence and provenance needed to consume them;
  • compact verification state keyed to content hashes;
  • dependency and compatibility metadata.

Runtime bundle excludes

  • source Python and test dependencies;
  • every render, screenshot, CI log and oracle diff;
  • live stock, price and distributor offers;
  • generated CAD tooling or regenerated model output;
  • parts the consuming project did not use.

Library releases contain the whole published catalogue. Project bundles vendor only the transitive closure of selected parts. The same content digest can appear in many projects without duplication of meaning. Updating a source dependency creates new compiled outputs; previous CompiledBoard artifacts remain untouched.

Sourcing stays separate. Manufacturer identity and approved exact alternates belong to the catalogue. Distributor SKU, price, stock and quote time remain project-owned sourcing snapshots, so market churn cannot change a canonical part hash.
10 · Independent verification

“Testable on its own” is a publish contract.

Normal pytest/Catch2 tests remain transparent author-authored evidence. Native gates prove that the shipped artifact, not merely the source declarations, carries the promised meaning.

Deterministic build

Build source twice in clean environments; manifests, canonical entries and asset digests match byte-for-byte.

gate · digest equality

Native reopen and round-trip

Open the emitted bundle through public C++, serialize it again and compare canonical meaning without importing Python.

gate · no authoring runtime

Structural integrity

Reject malformed units, inverted ranges, duplicate keys, dangling refs, bad terminal/pad mappings, missing asset bytes and digest mismatches.

gate · invalid state impossible

Electrical completeness

Run the versioned contract for each device class; prove required typed facts exist and exact-part capabilities satisfy fixture intent.

gate · typed meaning preserved

Projection lineup

Every logical pin appears correctly in symbols and maps to intentional footprint pads; 3D and optional simulation terminal transforms line up.

gate · cross-domain consistency

Representative fixture builds

Instantiate public entries through real bindings and compile Circuit, Schematic, multiple Boards, CompiledBoard, BOM and CPL outputs.

gate · end-to-end consumption

Visual and independent oracle review

Render symbols, footprints and 3D placement; compare risky geometry against an independently derived witness. Automation assists but does not replace review.

gate · geometry confidence

Release evidence

Record provenance, licence, model fidelity, render review and hardware-validation attestations against immutable digests.

gate · inspectable trust
Publishable versus loadable. Malformed artifacts fail at the native read boundary. Incomplete but structurally valid parts stay inspectable and emit diagnostics. The official catalogue’s release policy promotes selected diagnostics to blocking publish gates without making bad design intent structurally unloadable.
11 · Migration

Ten bounded slices. Electrical fidelity first.

Do not build a polished archive around today’s dual semantics. Close the meaning gap, establish one native path, then migrate real callers and delete compatibility surfaces only after parity.

00

Freeze vocabulary and observable behaviour

Snapshot native/public APIs, all four authoring routes, diagnostics, hashes, benchmark outputs and the v4 golden artifact.

Stop gate: exact-main inventories and reproducible fixtures are checked in.
01

Close electrical-fidelity gaps

Add stable native electrical contracts; lower exact values, tolerances, ranges and ratings through Python into PartDefinition; complete dimension coverage.

Stop gate: source → artifact → native reopen → selection preserves every typed attribute, and ERC consumes reopened data.

Add the native library aggregate

Introduce PartLibraryBuilder, manifest model, stable keys, duplicate checks, public query and explicit exact resolution.

Stop gate: C++ alone can build, validate and query an in-memory catalogue.

Stop gate: missing or mismatched referenced bytes reject, and no metadata-only hash remains.

Stop gate: source build and prebuilt reopen produce identical bundle and entry digests.
05

Stop gate: imported source and opened bundles reach one C++ path with module parity and lifetime tests.

12 · Boundaries

The direction is settled. Implementation details remain reviewable.

The user-level architecture no longer needs an A/B/C choice. Remaining decisions should be resolved per bounded slice without reopening ownership.

Settled by this review

  • Visible authoring, service and artifact planes.
  • C++ owns part meaning, validation, codecs, loading, query, resolution and artifact construction.
  • Python is a concise declaration and typed-handle surface.
  • Generic ComponentDefinition and exact-MPN PartDefinition stay distinct.
  • Typed electrical characteristics are first-class canonical part data.
  • Optional simulation models are separate hashed projections.
  • Queries return candidates; persisted design state pins an exact part digest.
  • Personal/team and official libraries share one format and test contract.
  • Local/Git dependencies first; registry later and outside the kernel.
  • Projects snapshot only consumed parts and assets.
  • GLB is viewer-ready; STEP is optional interchange.
  • Verification attestations key to immutable part hashes.

Still open, with recommendation

  1. Physical bundle: deterministic archive for distribution plus exploded developer inspection.
  2. Python exports: generated/declared typed names as primary UX; dynamic find for discovery.
  3. Resolution syntax: explicit path/Git/package entries in volt.toml, exact digests in volt.lock.
  4. Versioning: library semver for humans, immutable digest per part and asset for integrity.
  5. Electrical vocabulary: versioned capability contracts per device class, beginning with passives and power parts.
  6. Alternate parts: refs to separately defined exact parts, not unverified bare MPN strings.
  7. Simulation payloads: exact first supported set and licensing policy remain separate design work.
  8. Official scope: small evidence-rich pilot, then expand from demonstrated demand.
  9. Local deviations: prefer derived local parts; allow only narrow manifest-visible instance overrides.

Explicitly unchanged

A

Kernel ownership

Circuit remains append-only logical owner. Schematic and Board remain presentation and physical projections; neither creates or reinterprets connectivity.

B

Part selection

Exact selected-part intent remains logical/design-owned, not Board-owned. BOM/CPL and manufacturing project from it.

C

Project architecture

One project may produce multiple named Boards. Each CompiledBoard is an immutable historical artifact keyed by its inputs.

D

Export posture

KiCad and other outputs remain selected adapters. Native artifacts stay authoritative; no exporter or viewer becomes an alternate part owner.

E

Runtime discipline

No CAD generator executes during ordinary project consumption. No registry, live distributor service or package manager enters the kernel.

F

Change control

This page is a proposal and companion review. It changes no accepted ADR, product code, issue state or GitHub state.
