Volt architecture spec

Project Framework

Date: 2026-06-02

Status: proposed design and implementation plan

Scope: Python project structure, deterministic build result, declared assets, viewer bundle, and future analysis hooks.

Guiding sentence: A Volt project is a deterministic Python entrypoint for constructing kernel-owned electronics state and declared physical assets.

Thesis

volt.Project should be boring. It is not an application framework, package manager, task runner, or alternate EDA model. It is a tiny structure around the Python code that defines electronics, runs that definition, and returns a deterministic result that the app, viewer, tests, exporters, agents, and future simulation tools can consume.

The project framework gives Volt a stable answer to: "What code builds this board?" It does not change the answer to: "Who owns electronics meaning?" The kernel still owns circuits, schematics, PCB projections, validation, and serialized design state.

Problem

What is missing today

Current examples work by convention: each file defines a local build_example(), local artifact dataclasses, and local write_artifacts(...) functions. That is enough for tests, but it leaves every consumer to guess how to run a design, which outputs matter, where assets live, and how diagnostics should be collected.

Application problem

The future app needs one known entrypoint to run, inspect, visualize, and rerun after an agent changes code.

Viewer problem

The 3D viewer needs PCB JSON plus generated or referenced model assets, not only arbitrary Python state.

Simulation problem

Future analyses need a structured place to be declared without making simulation scripts own circuit meaning.

Without a project layer, every tool becomes a one-off runner for arbitrary Python. That is flexible in the worst way: powerful, but hard for an app, agent, CI job, or viewer to understand reliably.

Principles

Design principles

1. Project is orchestration, not EDA meaning

A project may call Python APIs, register generated assets, and collect outputs. Nets, components, footprints, schematics, board geometry, routes, rules, and diagnostics remain kernel-owned model data.

2. Boring primitives first

The core API is just project, build context, result, artifact, and asset. Higher-level templates and app workflows can compose those primitives later.

3. Python builds, JSON communicates

The app can execute project Python in a controlled run. The viewer should consume the resulting JSON and assets, never depend on executing arbitrary Python.

4. Determinism is a feature

Re-running the same project should produce stable artifacts, stable IDs, stable asset paths, stable hashes, and comparable diagnostics.

5. App hooks are downstream

The product app can watch files, run agents, show viewers, diff results, and manage UI state. The project framework only provides the reliable electronics construction contract.

6. Plain scripts still work

Volt remains a Python library. Project is the recommended structured path for serious, app-backed, bundled, or testable designs, not a requirement for every quick script.

Non-goals

What this must not become

The main failure mode is letting Project expand until it becomes a hidden application runtime. That would make Volt harder to understand and would blur the kernel ownership boundary.

  • No package manager or dependency resolver.
  • No general-purpose task graph.
  • No build system replacement.
  • No persistent workspace database.
  • No UI state store.
  • No hidden source of truth for circuit, schematic, PCB, simulation, or manufacturing semantics.
  • No Python-only EDA meaning that cannot be serialized or inspected after the run.
  • No requirement that every Volt script must become a project.
Layers

Ownership boundaries

Layer Owns Does not own
Kernel EDA state: components, pins, nets, physical parts, schematic projections, PCB projections, rules, validation, serialization. How a Python file is organized, where assets are written, app UI state, file watching.
Python authoring Ergonomic syntax for creating kernel-owned state and declared generated assets. Alternate semantics that cannot lower into the kernel or project result.
Project framework Build entrypoint, context, output collection, asset registry, diagnostics aggregation, deterministic bundle writing. Nets, routes, boards, simulation math, manufacturing meaning, app behavior.
Application Chat, editor, file watching, agent workflow, viewer UI, permissions, project navigation, result diffing. Canonical electronics meaning.
Viewer/exporters Projections from project result artifacts into 2D, 3D, KiCad, Gerber, STEP, simulation results, reports. Repairing or inventing missing design semantics.
API

First user-facing API

The first API should feel nearly invisible. A user names a project, marks the function that constructs the electronics, and then asks Volt to run it. 

import volt

project = volt.Project("my-board")

@project.build
def build(ctx: volt.BuildContext) -> volt.Design:
    design = volt.Design("my-board")

    # Define logical circuit.
    # Author schematic projection.
    # Author PCB projection.
    # Declare or generate physical assets when needed.

    return design

if __name__ == "__main__":
    project.write("dist/my-board.volt")

project.write(...) is convenience. The core operation is result = project.run(). A result is only the current output of executing the electronics definition. It is not version history, a cache, or persistent project state.

Types

Core data structures

Project

Names the project and stores one build entrypoint. It can run the entrypoint and write the result bundle.

BuildContext

Supplies deterministic paths, project metadata, asset registration helpers, and optional publish helpers.

ProjectResult

Simple immutable result of one run: design, collected projections, diagnostics, artifacts, and assets.

ProjectArtifact

A typed output such as logical JSON, PCB JSON, schematic JSON, SVG, validation report, or export report.

ProjectAsset

A referenced or generated file needed by artifacts, such as a 3D model, texture, STEP file, or future simulation dataset.

ProjectManifest

Deterministic index describing every artifact, asset, hash, role, and primary viewer entrypoint in the bundle.

class Project:
    def __init__(self, name: str, *, version: str = "0.1.0"): ...
    def build(self, fn): ...
    def run(self, *, output_dir: str | Path | None = None) -> ProjectResult: ...
    def write(self, path: str | Path) -> ProjectResult: ...

class BuildContext:
    project_name: str
    root: Path
    output_dir: Path | None
    assets: AssetRegistry
    def publish(self, item, *, role: str | None = None, name: str | None = None) -> None: ...

class ProjectResult:
    design: volt.Design
    artifacts: tuple[ProjectArtifact, ...]
    assets: tuple[ProjectAsset, ...]
    diagnostics: volt.DiagnosticReport
    def write(self, path: str | Path) -> None: ...
Return

What the build function returns

The first version should allow the build function to return a volt.Design. That keeps authoring simple and avoids inventing a project-specific EDA object. The result builder can discover projections attached to the design, then collect logical, schematic, PCB, SVG, and validation artifacts.

Default path

@project.build
def build(ctx):
    design = volt.Design("controller")
    ...
    return design

Explicit publish path

@project.build
def build(ctx):
    design = volt.Design("controller")
    schematic = design.schematic("Main")
    board = design.board("Main")

    ctx.publish(schematic, role="main")
    ctx.publish(board, role="main")
    return design

ctx.publish(...) is only an output selection primitive. It does not make the context own the board or schematic. It tells the project result which kernel-owned projections should be primary artifacts when automatic discovery is ambiguous.

Run

Project run lifecycle

LoadImport project Python
RunCall build(ctx)
CollectDesign, projections, diagnostics
LowerArtifacts and assets
WriteBundle or in-memory result
1

Create context

Resolve project root, deterministic output path, asset registry, and project metadata.

2

Execute build entrypoint

Call the registered function. Kernel mutation APIs reject structurally invalid state as usual.

3

Collect kernel outputs

Find the returned design and any published or discoverable schematic and PCB projections.

4

Validate and aggregate diagnostics

Run logical validation, schematic readiness/readability where applicable, PCB validation, and asset validation.

5

Materialize artifacts

Create deterministic JSON, SVG, reports, model assets, and a manifest that indexes the result.

Bundle

Viewer and application bundle

The result can be consumed in memory by tests and the app, or written as a directory bundle. The bundle is the handoff between Python execution and consumers that should not execute Python, such as the viewer. 

my-board.volt/
  manifest.volt.json
  logical/
    my-board.volt.json
  schematic/
    main.volt.schematic.json
    main.svg
    pages/
      main.svg
  pcb/
    main.volt.pcb.json

    report.json

The exact directory names can change, but the contract should not: the manifest lists all artifacts and assets, their roles, stable IDs, media types, paths, and hashes.

Manifest

Manifest shape

{
  "schema": "volt.project.manifest",
  "version": 1,
  "project": {
    "name": "my-board",
    "source": "project.py",
    "build": "build"
  },
  "entrypoints": {
    "logical": "logical/my-board.volt.json",
    "schematic": "schematic/main.volt.schematic.json",
    "pcb": "pcb/main.volt.pcb.json",
    "viewer": "pcb/main.volt.pcb.json"
  },
  "artifacts": [
    {
      "role": "pcb-json",
      "path": "pcb/main.volt.pcb.json",
      "media_type": "application/vnd.volt.pcb+json",
      "sha256": "..."
    }
  ],
  "assets": [
    {
      "role": "model-3d",
      "path": "assets/models/soic8.glb",
      "media_type": "model/gltf-binary",
      "sha256": "...",
      "source": "generated"
    }
  ],
  "diagnostics": {
    "path": "diagnostics/report.json",
    "summary": { "errors": 0, "warnings": 1, "infos": 0 }
  }
}

The PCB JSON can reference assets/models/soic8.glb. The manifest proves the asset exists and gives the viewer a hash and media type. This keeps binary model data out of normal JSON while still making the bundle complete.

Assets

Asset registry

Assets are declared during the project run, then copied or generated into the bundle. This is the missing bridge for code-defined or externally sourced physical models. 

@project.build
def build(ctx):
    design = volt.Design("my-board")

    soic_model = ctx.assets.generated(
        "models/soic8.glb",
        kind="model-3d",
        media_type="model/gltf-binary",
        write=lambda path: make_soic8_model(path),
        source="generated:parametric"
    )

    part = volt.PhysicalPartSpec(
        manufacturer="Texas Instruments",
        part_number="NE555DR",
        package="SOIC-8",
        footprint=SOIC8,
        pin_pads={...},
        properties={
            "model_3d": soic_model.ref
        },
    )

    return design

Generated assets

Created during the run from deterministic Python functions, package dimensions, or future CAD helpers.

Referenced assets

Existing vendor or library files copied into the bundle or referenced with stable URI and hash.

First implementation can store asset refs in structured properties until first-class 3D model value types land. The framework still gives the app and viewer a place to find the asset and verify it.

3D

How this supports the 3D viewer

The viewer should not execute Python. The project run is where Python-authored shapes, generated models, and external assets are resolved into a stable result. The viewer receives the PCB JSON plus manifest-indexed assets.

Python source: defines parts, footprints, board geometry, and declared or generated physical assets.

Project run: executes the source once, writes or indexes generated assets, and emits PCB JSON.

PCB JSON: references component model assets, placement transforms, board geometry, layers, and stable IDs.

Manifest: tells the viewer where those assets are and verifies hashes.

Viewer: loads JSON and assets, builds a 3D scene, and maps selections back to stable Volt IDs.

This lets Volt support Python-defined physical models without putting Python in the viewer. Code is the source of truth for authoring; JSON and assets are the portable projection for rendering.

Simulation

Future simulation hooks

Simulation should not be part of the first implementation, but the project layer gives it a natural future home. The project can declare named analyses while the kernel and simulation model contracts own the electrical meaning. 

@project.simulation("startup")
def startup(ctx, design):
    return volt.SimulationScenario(
        board="Main",
        duration="10ms",
        probes=["OUT", "+5V"],
    )

This is not a task runner. It is a way to expose "these are the analyses this electronics project defines" to the app, CI, and agents. The simulation graph, validation rules, and backend adapters remain separate kernel/model work.

App benefit

The app can list named analyses and run them without reading arbitrary script conventions.

Agent benefit

Agents can modify a circuit, rerun the project, and compare simulation outputs deterministically.

Kernel boundary

Project declares the scenario. Simulation contracts lower the design into analysis data.

Enables

What this enables

One-click project open

The app can load one file, find the build entrypoint, run it, and open the main board or schematic.

Reliable agent loop

Agent edits code, project reruns, result diff shows logical, schematic, PCB, 3D, and diagnostic changes.

3D viewer bundle

PCB JSON and model assets travel together, so the viewer gets complete render inputs.

Generated physical assets

Part definitions can generate model assets during the run and reference them deterministically.

Project regression tests

Tests can assert that a project run produces stable artifacts and expected diagnostics.

Manufacturing handoff

KiCad/Gerber/STEP exporters can consume a ProjectResult instead of re-running custom example code.

Simulation scenarios

Named analyses can become first-class project outputs later without becoming the project framework itself.

Component library QA

Library packages can include tiny projects that prove parts lower into footprints, board geometry, and assets.

Templateable projects

The app can create starter projects with normal Python files and a boring known entrypoint.

Tradeoffs

What we give up

  • Some arbitrary script freedom in the app-backed path. Users can still write scripts, but the product path has a known project entrypoint.
  • Build functions need to be deterministic enough for stable artifacts, hashes, and agent diffs.
  • Generated outputs become generated outputs. Users should edit source code, not hand-edit bundle artifacts.
  • The framework adds a small API surface that must be maintained with the same care as the rest of Python authoring.
  • Application runners will need to execute project Python, which is powerful but not a safe declarative file format. The bundle is the safer viewer/export surface.
  • We will need rules for asset path collisions, stale generated assets, and what happens when a build fails halfway.

These are acceptable tradeoffs because they buy a stable boundary between source code and product experience. The alternative is every app feature inventing its own way to run and understand arbitrary Python files.

Errors

Error handling and diagnostics

Structural errors

Invalid kernel mutations still fail immediately. The project framework should not catch and reinterpret invalid kernel state as diagnostics.

Design diagnostics

Logical validation, schematic readiness/readability, PCB DRC, asset missing warnings, and future simulation readiness checks are aggregated into result diagnostics. 

{
  "build": {
    "status": "completed",
    "errors": 0,
    "warnings": 2
  },
  "reports": {
    "logical": "diagnostics/logical.json",
    "schematic:Main": "diagnostics/schematic-main.json",
    "pcb:Main": "diagnostics/pcb-main.json",
    "assets": "diagnostics/assets.json"
  }
}

If Python raises during the build, the app receives a project run failure with traceback context. If the design builds but has bad design intent, the result contains diagnostics.

Testing

Testing strategy

Unit tests

Project registration, duplicate build functions, invalid return values, asset registration, manifest sorting.

Example tests

Migrate LED board and 555 benchmark to project runs while preserving deterministic output.

Bundle tests

Write the same project twice and assert identical manifest, JSON, SVG, diagnostics, and asset hashes.

Failure tests

Missing build function, build exception, missing asset writer output, duplicate asset path, invalid artifact role.

Viewer contract tests

Assert PCB JSON model refs are present in manifest assets and every referenced asset exists.

Compatibility tests

Plain scripts and existing direct Design, Schematic, and Board APIs continue to work.

Plan

Implementation plan

1

Introduce the minimal project API

Add Project, BuildContext, ProjectResult, ProjectArtifact, and ProjectAsset in Python. First version supports one build function returning Design.

2

Collect existing outputs

Convert logical JSON, schematic JSON/SVG, PCB JSON/SVG, and validation reports into result artifacts. Use direct existing APIs rather than adding new kernel semantics.

3

Add deterministic bundle writing

Write a directory bundle with manifest, artifacts, diagnostics, and asset entries. Add stable ordering and hash calculation.

4

Add asset registry

Support referenced and generated assets. Begin with generic file assets, then connect this to PCB 3D model refs when those value types are settled.

5

Migrate examples

Convert PCB LED board and 555 timer board examples to volt.Project while preserving old helper functions temporarily if tests or docs still call them.

6

Define application integration contract

Document how the app locates a project file, runs it, reads a result, opens the viewer, and maps selected IDs back to source-facing objects.

7

Add future analysis registration only after simulation contracts

Keep simulation hooks out of the first PR unless they are only documentation. Add them after kernel-owned simulation model contracts exist.

Questions

Decisions to make before implementation

  1. Should project.write(path) write a directory bundle by default, or should it require write_bundle(...) for clarity?
  2. Should the first version auto-discover all boards and schematics, or require explicit ctx.publish(...) for the primary outputs?
  3. Should bundle paths include project name in filenames, or use stable generic paths such as pcb/main.volt.pcb.json?
  4. Should build functions be required to accept ctx, or should zero-argument build functions be supported for small scripts?
  5. Should generated assets be written during build execution, or staged and written only when the result bundle is written?

Recommended defaults: directory bundle, auto-discover with optional explicit publish, project-name filenames for human readability, require ctx, and stage generated assets until bundle writing so failed runs do not leave partial output.

Definition

Definition of done for the first milestone

  • volt.Project can register exactly one build function.
  • A project run returns a ProjectResult containing the returned Design.
  • The result can write deterministic logical, schematic, PCB, SVG, diagnostics, manifest, and asset entries for existing examples.
  • The result exposes artifacts and assets in memory for the application.
  • Generated output paths are deterministic and reject unsafe or duplicate paths.
  • Existing direct Python authoring APIs remain valid.
  • At least one PCB example and one schematic-only example use the framework.
  • Tests prove repeated runs are stable.
