Project Framework v1 Policy Decisions

This note closes the remaining policy questions for the first staged Project Framework release. The through-line is simple: the framework may help author, test, and package work, but it does not become a second owner of EDA meaning. Returned Volt models stay canonical. BuildContext and the project-result bundle stay authoring and delivery surfaces around those models.

Stable Input Rule

Later stages receive BuildContext every time. Adding a resource or a second design must not silently change the parameter type seen by downstream code.

Status Boundary

ProjectResult.status and the written bundle summarize one run for tools, CI, and the app. They are not persistent project state and not a kernel database.

Honest Naming

“Project Framework” is the accurate term. v1 currently ships a staged flow, but the intent is an authoring framework around kernel-owned models, not a permanently fixed pipeline dressed up as one.

Decision Summary

Topic Decision
Multi-model stage tests Attached @stage.test helpers remain a single-model feature in v1. When a stage returns multiple models, the framework reports that those attached tests are unsupported instead of guessing a per-model policy.
Missing expected diagnostics A declared expected diagnostic that does not appear is a failure. Stale expectations are policy drift, not success.
Bundle and status ownership Python owns project-result bundling and run status reporting. The bundle is a deterministic handoff artifact, not canonical EDA state.
Framework naming Use “Project Framework” or “staged Project Framework,” not “fixed pipeline,” when describing v1 behavior and roadmap intent.

1. Multi-Model Stage-Test Policy

v1 keeps the attached test API intentionally narrow. The check helpers are ergonomic because they operate on one design, one schematic, or one board at a time. Once a stage returns multiple models, the framework should not invent implicit iteration, fan-out naming rules, or aggregate pass/fail semantics behind the user’s back. Instead, it should stay explicit and report that attached tests are unsupported for multi-model stage outputs.

Use ordinary pytest over ProjectResult for cross-model assertions in v1. That keeps policy obvious and avoids baking accidental semantics into @stage.test.

2. Missing Expected Diagnostics Policy

Expected diagnostics are active product policy, not comments. If a project declares an expected diagnostic and the run no longer emits it, the framework should treat that expectation as missing and keep expected_diagnostics_ok false. This catches stale waivers, renamed reports, and unreviewed intent drift.

The important distinction is between “the design still has a known issue” and “the policy file still says it should.” v1 keeps that distinction sharp.

3. Python-Owned Bundle and Status Boundary

The project-result bundle, manifest, diagnostics JSON, test JSON, and top-level run status belong to the Python framework surface. They package one run so other tools can inspect it without re-executing arbitrary Python. That makes them a delivery contract, not a second source of circuit truth.

Kernel-owned meaning still lives in the returned Volt models and their serialized logical, schematic, and PCB data. The framework may summarize that output as clean, expected-diagnostics, or failed, but those labels are run reporting, not new EDA semantics.

4. Framework vs. Fixed-Pipeline Naming Honesty

v1 happens to execute a fixed set of named stages, but that does not make the concept a “fixed pipeline” in product language. The value being introduced is a framework for staged authoring, validation, and deterministic handoff around kernel-owned models. Calling it a fixed pipeline overstates rigidity and understates the long-term role of project-level authoring structure.

When describing today’s implementation, say “staged Project Framework” if the execution order matters. When describing the product surface, say “Project Framework.”

Follow-Ups Left Out of This PR

  • If teams want attached multi-model stage tests, add a separate explicit API for that instead of overloading the current single-model helpers.
  • If bundle consumers need richer machine states, add derived manifest fields rather than widening the core three-state status vocabulary.
  • If future work adds custom stage graphs beyond design, schematic, and board, model that capability directly rather than retroactively naming v1 a pipeline.
