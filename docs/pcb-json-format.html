Format v1

Volt PCB projection data is stable board state plus recomputable viewer affordances.

Format: volt.pcb

Version: 1

Units: millimeters for all board and footprint geometry

Scope: placement-ready PCB projection over one logical circuit

Boundary

The kernel owns PCB meaning. JSON and SVG are projections.

The PCB JSON document stores enough board geometry, semantic IDs, layer metadata, footprint geometry, placements, and logical references for a downstream application to render and inspect a board without reading the SVG preview. The SVG is a deterministic preview with matching selectors where practical.

Canonical

Persisted board projection

board data is loaded into volt::Board: name, units, layers, stack, outline, features, footprint definitions, and placements.

Derived

Viewer cache

viewer.pad_resolutions and viewer.diagnostics are validated when present, then recomputed from kernel-owned board and circuit state.

Consumer-owned

Application behavior

Clicks, selection state, editor workflows, prompts, and 2D or 3D viewer behavior belong to the consuming application.

Document Shape
{
  "format": "volt.pcb",
  "version": 1,
  "board": {
    "id": "board:0",
    "name": "Control",
    "units": "mm",
    "layers": [],
    "layer_stack": null,
    "outline": null,
    "features": [],
    "footprint_definitions": [],
    "placements": []
  },
  "viewer": {
    "pad_resolutions": [],
    "diagnostics": []
  }
}

Field order is deterministic. Entity arrays use insertion order. Local IDs are stable within a document and use typed prefixes such as board_layer:, board_feature:, footprint_def:, footprint_pad:, component_placement:, and pcb_pad:.

Layers and features do not define connectivity.

They describe board projection geometry and metadata. Logical pins and nets remain owned by the logical circuit and are only referenced by ID.

Renderer Use

A renderer should consume JSON directly.

The SVG preview is optional debug output. A 2D or first 3D renderer should use the JSON board outline, thickness, features, footprint pad geometry, placement transforms, and pad projection identity as its input model.

Renderer primitiveJSON inputsContract
Board bodyboard.outline.vertices, board.layer_stack.board_thickness_mm, board.unitsExtrude the outline by board thickness in millimeters. The renderer chooses camera and z-origin conventions.
Layer controlsboard.layers[], board.layer_stack.layers[]Use layer IDs, names, roles, sides, enabled state, and stack order for visibility and inspection.
Mechanical featuresboard.features[]Render board holes and future board features from stable feature IDs and board-space geometry.
Footprint pad bodiesboard.footprint_definitions[].pads[]Render pad shape, size, drill, and layer membership in footprint-local coordinates.
Placed padsboard.placements[] plus referenced footprint_defApply placement side, rotation, and translation to every footprint-local pad. Bottom-side placements mirror local X before rotation.
Selection identityviewer.pad_resolutions[], viewer.diagnostics[]Use derived pad projection IDs, pin refs, net refs, and diagnostic entity refs for overlays and inspection panels.

Routed copper, vias, zones, rich package bodies, component heights, and external 3D model references are intentionally absent from v1. They should be added as kernel-owned PCB projection data when those concepts exist in Volt.

SVG Preview

SVG selectors mirror JSON identity where practical.

The preview is deterministic output for inspection and snapshot testing. It is not the canonical board format, but downstream viewers can use its data attributes to correlate visible shapes with JSON entities.

Load Checks

Malformed structure fails at the load boundary.

The reader rejects duplicate or non-sequential IDs, dangling layer-stack entries, placements that reference missing components, footprint references that do not match selected physical parts, stale pad projection caches, dangling pin or net refs, and malformed diagnostic entity refs.

  • Structural integrity is enforced by the reader and kernel mutation APIs.
  • Bad design intent, such as missing placement data or off-board pads, is reported through diagnostics.
  • Viewer cache values must match recomputed kernel-owned state; the cache is never trusted as a source of truth.
Non-goals
  • No product UI, 3D renderer, click payload, editor workflow, or agent feedback protocol is defined here.
  • No source section, source-code location mapping, or viewer-to-agent source trace is part of v1.
  • No direct UI mutation of kernel state is authorized by this data.
  • No routing, autorouting, DRC, Gerber, KiCad PCB import/export, or manufacturing output contract is implied.
