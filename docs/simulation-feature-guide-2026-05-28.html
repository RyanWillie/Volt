Feature guide / May 28, 2026

Simulation should become Volt's executable circuit contract.

The feature is not a Python-only update loop and not a SPICE wrapper bolted onto schematics. It is a kernel-owned analysis layer that lowers the logical circuit into solver-ready models, lets Python orchestrate scenarios, and turns circuit behavior into assertions a design can carry over time.

Read this first
Recommended stance: Volt simulation should start from the canonical logical circuit, consume selected parts and typed electrical semantics, produce a backend-neutral simulation graph, and expose Python-native scenarios, probes, assertions, and reports. SPICE should be an adapter/backend. PCB geometry should later enrich the graph with parasitics.

What this enables

Users can ask whether a circuit behaves within intent, not merely whether pins are connected: LED current, rail sag, regulator load, reset timing, filter response, load steps, PWM behavior, and module contracts.

The architectural correction

Components can be behavior-rich, but nets do not belong to component code. Models contribute equations, sources, or discrete state. The solver computes the mutually consistent board state. 

Volt Circuit
  -> simulation readiness diagnostics
  -> backend-neutral SimulationGraph
      -> SPICE export / ngspice or Xyce runner
      -> Python scenario and behavioral runtime
      -> future internal MNA solver
  -> SimulationResult
      -> probes, plots, assertions, regression tests, reports
Product idea

Make circuits testable.

The user-facing promise should be simple: define your circuit in Volt, then run executable checks against the electrical behavior you care about. A user should be able to swap an LED, change a resistor, replace a regulator, or add a subcircuit and know which behavioral contracts still hold.

Explore

Inspect voltages, currents, power, device states, and thresholds after an operating-point solve or during a transient scenario.

Assert

Encode intent as checks: rail limits, LED current bounds, GPIO current limits, regulator dissipation, startup timing, and filter behavior.

Regress

Run simulation cases in CI so circuit edits, component substitutions, and module reuse do not silently break known-good behavior. 

def test_status_led_contract():
    case = design.simulation.case("status_led_on")
    case.drive(mcu.pin("PA5"), "high")

    result = case.dc_operating_point()

    result.assert_between("3V3.voltage", "3.20V", "3.40V")
    result.assert_between("D_STATUS.current", "2mA", "8mA")
    result.assert_less("U_REG.power", "500mW")

This is intentionally closer to software testing than traditional schematic probing. The circuit still uses real solvers, but the workflow is shaped around repeatable design intent.

Mental model

AFSIM-like scenarios over solver-owned physics.

The scenario layer can feel like an entity simulation: devices wake up, GPIOs change state, switches close, loads appear, and probes observe the result. The difference is that electrical state is a shared physical solve, not a value each component updates independently.

1. Scenario Events change model state: button press, GPIO high, PWM duty, load step.
2. Models Components contribute sources, equations, impedances, or discrete drive modes.
3. Solver The backend solves the whole circuit state for this time or frequency point.
4. State Volt records node voltages, branch currents, component power, and model states.
5. Checks Python probes, assertions, logs, and plots inspect the solved snapshot.
Important: an LED turn-on scenario is usually DC operating-point or transient analysis, not AC. AC analysis is small-signal frequency response around an operating point. Volt should make that distinction visible in the API.

Analyses

Use the standard analysis vocabulary, but wrap it in Volt intent.

Operating point

One solved steady-state snapshot. This should be the first practical target: rail voltages, LED current, input current, regulator power, GPIO current.

DC sweep

Vary one or more sources or parameters and check ranges: VIN tolerance, load current, resistor choice, threshold margins, component substitutions.

Transient

Time-domain behavior: startup, RC timing, PWM, load steps, button press, capacitor charge, ringing, and rail dip.

AC small-signal

Frequency response around a solved operating point: filters, gain, phase, impedance, and loop-shape style questions.

Later analyses

Noise, sensitivity, pole-zero, Monte Carlo, corners, and statistical part variation should wait until model contracts and result artifacts are stable.

Not an analysis

"Update every component once" is not a valid analog solve. It can be useful for digital/event behavior, but not for arbitrary electrical networks.

Existing simulators validate this vocabulary. KiCad exposes AC sweep, DC transfer, operating point, and transient modes through ngspice, and lets users probe net voltages and pin currents. Ngspice itself supports DC, AC small-signal, transient, noise, sensitivity, and related analyses.

References: KiCad SPICE overview, ngspice supported analyses.

Architecture

Kernel-first simulation architecture.

Simulation should be a sibling analysis layer to schematic and future PCB projections. It consumes circuit meaning; it does not create connectivity. Runtime results are analysis output; they are not design fields.

Core
Simulation contracts

Define model identity, supported analyses, terminals, parameter schema, required typed attributes, fidelity level, assumptions, and provenance. Contracts are kernel-owned so Python, JSON, validation, and adapters see the same simulation meaning.

Core
Simulation graph

Lower the canonical circuit into nodes, terminals, element instances, sources, probes, assertions, and model references. This graph is backend neutral and should be printable as a debug artifact before any solver runs.

Adapter
SPICE export and runner

Export supported graph subsets into SPICE netlists and optionally run ngspice or Xyce. Unsupported devices produce explicit diagnostics. SPICE is an adapter, not Volt's canonical model.

Adapter
Python scenario runtime

Python schedules events, changes behavioral model state, defines stimuli, requests analyses, and inspects result snapshots. Python does not own the electrical state and should not silently create simulation semantics that the kernel cannot serialize.

Later
Internal solver

A small modified-nodal-analysis solver may become worthwhile after contracts, graph lowering, and SPICE export prove the model. Start with linear DC and simple transient, not full SPICE replacement.

Later
PCB-aware enrichment

Once PCB geometry exists, extract trace resistance, via resistance, plane impedance, capacitance, inductance, or transmission-line approximations into additional graph elements. Logical simulation comes first.

Models

Model fidelity should be explicit.

Volt should not pretend all simulated components are equally trustworthy. Every model attachment should state its supported analyses, approximation level, and limitations. This gives users a practical path from simple educational circuits to vendor-model-backed design checks.

Model level Examples What Volt should report
Ideal Ideal voltage source, ideal switch, ideal resistor, fixed LED drop. Fast and useful, but result report should mark idealized assumptions.
Approximate GPIO output resistance, simple diode equation, LDO output impedance, battery internal resistance. Good for board-level design intent checks, not signoff.
Vendor/SPICE Manufacturer diode, op amp, regulator, MOSFET, or subcircuit model. Best external-model path, with adapter compatibility and license/provenance notes.
Behavioral MCU pin logic, PWM, load state machine, sensor stimulus, fault injection. Must expose solver contributions, thresholds, timing, and deterministic scheduling.

KiCad's SPICE integration highlights the model-availability problem: KiCad integrates ngspice, but third-party models usually come from manufacturers. Volt should make missing and approximate models first-class diagnostics instead of burying them in backend logs.

Python surface

Python should orchestrate, inspect, and declare contributions.

The Python API can feel expressive and scenario-driven while still preserving the kernel boundary. The key is that Python asks the solver to run with changed model state; it does not imperatively assign final net voltages. 

case = design.simulation.case("led_load_step")

case.drive(mcu.pin("PA5"), state="low")
case.at("5ms").drive(mcu.pin("PA5"), state="high")
case.probe("3V3.voltage")
case.probe(component("D_STATUS").current)
case.assert_between("3V3.voltage", "3.20V", "3.40V")
case.assert_less("PA5.current", "12mA")

result = case.transient(stop="20ms", step_hint="10us")
result.plot("3V3.voltage", "D_STATUS.current")

Good Python behavior

  • Drive GPIO high, low, high-Z, PWM, or open-drain.
  • Enable a load, close a switch, inject a pulse.
  • Read a solved snapshot and update internal model state.
  • Attach assertions and probes to named design objects.

Bad Python behavior

  • Set arbitrary net voltages after the solve.
  • Invent hidden connectivity or hidden simulation-only nets.
  • Depend on schematic labels as simulation identity.
  • Hide model assumptions outside serialized kernel data.

PySpice shows that Python-driven circuit simulation is practical, but Volt's advantage should be stronger: Python code operates over kernel-owned components, pins, nets, parts, and module contracts, not a separate script-only circuit.

Reference: PySpice documentation.

Netlist and PCB

Start from the logical netlist; later enrich from PCB.

The first simulation system should operate over the logical circuit: components, pins, nets, selected parts, typed attributes, constraints, models, sources, and probes. That is enough for classic SPICE-style simulation and most board-level intent checks.

Layer Simulation role When to build
Logical circuit Canonical connectivity, component instances, pins, nets, selected parts, typed values. First. This is the primary simulation source.
Schematic Probe placement, visualization, result overlays, and human navigation. After graph/results exist. It should not define simulation connectivity.
PCB Parasitic extraction: trace resistance, via resistance, capacitance, inductance, plane effects. After PCB model exists and has enough geometry to extract useful approximations. 
Logical simulation:
  Net 3V3 is one node.
  LED current and regulator load are solved from component models.

PCB-aware simulation:
  3V3 becomes regulator pad -> trace segment -> via -> MCU pad.
  Small R/L/C elements are inserted to model physical implementation.
External ecosystem

Existing tools prove the pieces, but not the Volt shape.

The proposed direction is possible because the pieces already exist in adjacent ecosystems. The opportunity is to combine them with Volt's kernel-owned circuit model and Python-native design workflow.

SPICE and KiCad

Proven electrical solving, standard analyses, vendor models, and schematic probes. The lesson: use SPICE compatibility early, but do not let SPICE text become Volt's canonical model.

PySpice

Proves Python can control ngspice/Xyce-style workflows. The gap: Python circuits are not usually tied to a first-class EDA kernel with schematic and future PCB projections.

SystemC AMS

Proves system-level mixed-signal modeling and virtual prototyping. The lesson: scenario and behavioral layers need explicit scheduling semantics.

Modelica and Simscape

Prove declarative, multi-domain component modeling. The lesson: model contracts and equations matter more than imperative per-component updates.

Renode and co-sim

Proves virtual platforms can coordinate high-level scenarios with lower level simulators. The lesson: adapter boundaries are normal and valuable.

Volt's opening

Use the logical circuit as the source, keep semantics typed, and make probes and assertions feel native rather than a simulator side quest.

References: SystemC AMS overview, Modelica Electrical, Simscape Electrical, Renode HDL co-simulation.

Readiness checks

Fail before solver failure.

The first great user experience is not a perfect solver. It is a readiness report that says exactly what must be modeled before an analysis is meaningful.

Required diagnostics

  • No reference node or ambiguous ground.
  • Power rail has no modeled source.
  • Component has no model for requested analysis.
  • Model terminals are not bound to component pins.
  • Required typed attributes are missing.
  • Unsupported idealization or backend feature.

Useful warnings

  • Ideal supply hides rail sag.
  • Approximate component model used.
  • Selected part lacks vendor model.
  • Open pins affect analysis result.
  • Simulation ignores PCB parasitics.
  • Backend convergence settings changed.

These checks should reuse and extend existing ERC work. VOL-71, VOL-72, VOL-73, and VOL-74 are the right backbone: kernel-owned simulation contracts, ERC prerequisites, graph lowering, and a SPICE adapter only after the graph exists.

Practical rollout

Build the feature in vertical slices.

  1. Simulation contracts and readiness: add kernel-owned model definitions, attachments, analysis specs, probes, and diagnostics. No solver yet. Verify with JSON round trips and readiness fixtures.
  2. Simulation graph lowering: lower simple logical circuits into a printable graph. Support resistors, capacitors, inductors, ideal sources, ground/reference, probes, and unsupported-model diagnostics.
  3. SPICE export subset: export the graph to a small SPICE subset with golden tests. Keep this as an adapter with explicit loss reports.
  4. Operating-point runner: optionally invoke ngspice/Xyce for simple fixtures and return normalized SimulationResult objects.
  5. Python probes and assertions: let Python users write design.simulation.case(...), run supported analyses, and assert over results.
  6. Behavioral scenarios: add event scheduling, GPIO drive modes, PWM, switch/load control, and stateful behavioral models that contribute solver inputs rather than owning net voltages.
  7. Module contracts: let reusable subcircuits carry simulation cases: input ranges, guaranteed outputs, load limits, startup timing, and selected-part compatibility expectations.
  8. PCB enrichment: after PCB foundation, extract simple parasitics and compare logical-only vs PCB-aware behavior.
Do not start by writing a full analog solver. Start by making Volt able to say what should be simulated, why a circuit is not ready, and how to lower supported meaning into a backend-neutral graph.
What to defer

Keep the first class feature focused.

Avoid
Python-only simulation semantics

A callback that secretly defines electrical meaning outside the kernel will break serialization, validation, review, and backend interoperability.

Avoid
SPICE as the internal schema

SPICE compatibility is valuable, but Volt should not encode its canonical model as text netlist quirks, backend-specific syntax, or vendor model escape hatches.

Avoid
Signoff claims

Early simulation should support design exploration and regression tests, not guarantee manufacturing correctness. Reports must state model fidelity and ignored effects.

Avoid
PCB effects before PCB data exists

Let logical simulation stand on its own. Add board-aware parasitics only when footprint placement, pads, routes, vias, and planes exist in the kernel.

Decision

The feature should feel native to Volt.

Volt's simulation should be presented as an executable extension of the circuit model: the same components, pins, nets, selected parts, modules, and typed attributes that define the design also define what can be analyzed. That is what makes the feature first class.

Planning recommendation: keep PCB foundation as the next main product track, but promote simulation contracts into a parallel design track now. The immediate simulation milestone should be VOL-71 plus VOL-72: model contracts and readiness diagnostics. The first implementation milestone should be VOL-73: lower a small, typed logical circuit into a simulation graph.

Once that graph exists, Volt can support both standard SPICE workflows and the more powerful Python-native promise: "here is my current circuit, check anything I care about, and tell me if this change broke the behavior I expected."
