Baseline testing review · 2026-05-16

Volt needed a test suite, not a giant LED story.

This review captured the pre-overhaul baseline that motivated VOL-112. The C++ tests were fast and well aligned with the kernel. The Python surface had useful coverage, but it was concentrated in one giant file that acted as example, regression pack, fixture library, and API contract all at once.

Implementation note
VOL-112 applied the first slice: the LED MVP file was removed, Python behavior was split into focused CTest-visible files, and the Python test entrypoint now uses the same interpreter that CMake used to build the Python extension. The rest of this document remains as the baseline review and trust model behind the work.
Verified baseline
243 CTest cases after rebuilding dev; all pass.
81 Python tests inside python/tests/test_led_mvp.py.
90.7% C++ filtered line coverage from cmake --workflow --preset coverage.
The important positive: the suite is quick. A rebuilt ctest --preset dev run completed in under a second locally, and the CI matrix already runs Linux, macOS, Windows, formatting, build config, tests, docs, and Linux coverage.
The important warning: before rebuilding, CTest failed because the build tree was stale. Direct pytest also failed because the shell had Python 3.13 while CMake built the extension for Python 3.14, and that 3.14 interpreter does not have pytest installed. Testing needs one blessed Python entrypoint.
Baseline shape
Area What exists Read
C++ kernel and projections 31 focused *_test.cpp files, one volt_core_tests binary, 239 Catch2 cases discovered into CTest. Mostly healthy. It maps well to architecture layers and the kernel principle.
Python facade 2 files: test_led_mvp.py with 81 tests and test_stm32_usb_buck_example.py with 3 tests. Too much behavior is hidden in one file; failures are harder to localize than they should be.
Examples C++ LED example is a CTest executable. Python STM32 USB buck example is an acceptance-style script. Good ingredients, but LED should become a small readable smoke example, not the regression archive.
Coverage Coverage preset disables Python bindings and reports C++ headers/sources only; threshold is 80% in CI. Useful signal, not sufficient trust. It misses Python facade behavior and binding glue.
Trust model
Kernel invariants Reject impossible structure at mutation and load boundaries: missing IDs, dangling refs, duplicate pin-net membership.
Diagnostics Report bad design intent without mutating structure: unconnected pins, single-pin nets, electrical role conflicts.
Serialization Guarantee deterministic write, parse, write cycles across logical circuit and schematic documents.
Python API Prove the facade is ergonomic but still kernel-owned: no Python-only EDA meaning, no hidden logical mutation by schematics.
Examples Keep examples as readable acceptance tests that show real usage and catch integration breaks.

This mirrors Volt's architecture: C++ owns EDA meaning, Python authors through it, schematic and future PCB layers project or implement existing logical state.

Recommended structure
python/tests/
  conftest.py                         # shared factories and JSON helpers, when needed
  test_design_authoring.py             # Design, nets, components, custom definitions
  test_component_library.py            # LibraryComponent, selected parts, pin mapping, STM32 catalog
  test_design_intent.py                # stub nets, no-connect intent, diagnostic suppression
  test_schematic_projection.py         # place/wire/label/power/no-connect projection JSON
  test_schematic_drawing.py            # cursor, two-terminal grammar, anchors, route shortcuts
  test_schematic_io.py                 # schematic JSON load/write/SVG smoke
  test_python_diagnostics.py           # DiagnosticReport and EntityRef inspectability
  examples/
    test_led_example.py                # clean LED example as documentation-quality smoke test
    test_stm32_usb_buck_example.py     # larger acceptance test, still kept separate

Keep CTest as the top-level contract

CMake already chooses the interpreter that built _volt. Every local and CI command should go through that same interpreter.

Keep pytest explicit

Pytest is now an explicit dev dependency. CTest runs ${Python3_EXECUTABLE} -m pytest so the Python facade is tested by the interpreter that built the extension.

Recommended work
Phase 1

Make the Python harness boring

Add one blessed Python test entrypoint in CMake. It must use the exact Python3_EXECUTABLE that built the extension, set PYTHONPATH to build/dev/python, and expose useful per-test failure names. Add CTest labels such as python, cpp, example, coverage, and integration.

Phase 2

Split test_led_mvp.py by behavior

Move tests without changing semantics. Create shared factories only after the first two splits reveal repeated setup. The LED example should shrink to one happy-path logical circuit test and one optional rendered schematic smoke test.

Phase 3

Add invariant and metamorphic checks

Add generated or table-driven cases for connect/disconnect idempotence, one-net-per-pin enforcement, schematic presentation operations preserving logical JSON, and logical/schematic read-write determinism. Keep random tests seeded and small enough for normal CTest.

Phase 4

Raise trust gates after the split

Ratchet C++ coverage by module instead of only the global 80% line gate, add Python facade coverage if pytest and coverage.py are accepted dependencies, and keep sanitizer runs on the C++ path. Do not chase 100%; chase missing behavior at ownership boundaries.

Specific gaps

Python failure isolation

CTest sees the giant LED file as one test. A regression in one Python behavior can hide the health of the rest of the facade.

Interpreter drift

The repo currently makes it easy to run Python tests with a different interpreter than the extension was built for. That is a sharp edge for contributors and agents.

Python coverage blind spot

The coverage preset disables Python, so python/volt/__init__.py and the binding experience are trusted only through pass/fail tests.

Fixture granularity

There are good C++ golden fixtures, but Python does many direct JSON shape assertions inline. Shared assertion helpers would make intent easier to read.

Example responsibility

LED should teach the API. STM32 can remain an acceptance test. Neither should be the only home for core Python behavior regressions.

Validation boundary proof

The suite should repeatedly prove the project rule: structural impossibility is rejected at mutation/load time, while design intent issues are diagnostics.

Definition of done
  1. cmake --workflow --preset dev passes from a clean build tree.
  2. ctest --preset dev -L python can run just Python tests with individual failure names.
  3. ctest --preset dev -L example runs the readable LED smoke test and the STM32 acceptance test.
  4. cmake --workflow --preset coverage still passes and reports no unexplained drop from the current 90.7% filtered C++ line coverage.
  5. The old test_led_mvp.py is gone or reduced to a compatibility shim with no unique assertions.
  6. The testing docs explain the interpreter rule so nobody reaches for the wrong pytest binary again.
Do not start by rewriting the Python API or the examples. This is a test-structure task first. Move assertions intact, preserve current behavior, then add missing trust checks in the new homes.
