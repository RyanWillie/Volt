Issue #269 · post-Circuit

Keep the owners.
Compile the handoff.

Circuit’s completed migration proved the value of a small typed kernel owner. The next step is not to press Schematic, Board, IO, Python, CLI and Project into one generic mould. It is to align each native owner, make the Board handoff complete, and let built artifacts be inspected without executing authoring code again.

01 · Decision brief

A selective redesign,
not a generic replay of Circuit

Adopt a typed projection + compiled artifact boundary: retain distinct Circuit, Schematic and Board authoring models; split their native binding ownership; compile Board once into a complete immutable snapshot; separate canonical models from reproducible views; and add a read-only ProjectBundle path.

Approved direction: Option C — owner-aligned projection aggregates plus a compiled Board and typed artifact graph.

This resolves the measured seams—49/50-declaration roots, a 133-method native façade, 13 hidden footprint-library injections, mixed PCB model/view JSON, and inspect commands that rerun Python—without inventing a universal Workspace, alternate netlist owner, or generic stage graph.

Do

Compress storage-shaped C++ reads, separate native owners, introduce CompiledBoard, classify artifacts, and open bundles read-only.

Keep

Current semantic ownership, fluent Python, fixed public stages, Python-owned run status/bundling, deterministic model formats, and native fabrication authority.

Defer

Generic user-defined stages, a public low-level Python model API, multiple native binaries, and format breaks until concrete evidence or a focused ADR justifies them.

Circuit boundary complete

Parity and final API/storage compression are complete.

#261 and #266 are complete. PR #288 finalized the aggregate at 15 callable operations and 16 declarations including the deleted-rvalue safety overload, with 6,681 production Circuit lines. Semantic, persistence, diagnostic, Python and CI parity remain proven under #267.

#269 remains open while focused ADRs and the implementation dependency graph are admitted. Option C, public C++ bundle loading, selected exports, CompiledBoard, and multiple named Boards are approved; v1 Boards are complete alternative physical implementations, not implicit assembly partitions.

Companion source of truth: the part-library architecture reference owns reusable component contracts, exact parts, canonical Voltage/Current semantics, native library bundles and integrity-bearing selection. The two references are intentionally separate and jointly constrain the full refactor program.

Coupling that should be removed

  • Projection mutation routed through a class named and owned as Circuit.
  • Exactly one optional Board enforced by binding state rather than a public product decision.
  • Board validation/render/export depending on lazy Python footprint cache synchronisation.
  • Model codecs, rendering, reports, interop and manufacturing concentrated in Volt::IO.
  • Project artifact writing implemented as hard-coded model-type branches.

Coupling that is intentional

  • Schematic and Board retain a live logical Circuit dependency while authoring.
  • Board routing and spatial algorithms operate over Board state.
  • Python keeps ergonomic registries and fluent builders, provided they lower to kernel meaning.
  • Project stages depend on design output through stable BuildContext.
  • Exporters depend on typed compiled inputs, not generic records.

The reopening path is missing.

05 · Circuit lessons

Generalise the discipline,
not the silhouette

Circuit’s completed semantic, persistence, diagnostic, Python and API-compression work supplies an operating method and ownership rules. Its small typed boundary, append-only graph shape and transaction semantics are not universal properties of every EDA projection or service.

Carry forward

  • One native semantic owner per domain.
  • Complete typed compound specs at mutation boundaries.
  • Structural invalidity rejected immediately.
  • Typed IDs and closed, domain-specific updates.
  • get/all over stable tables; free derived queries.
  • Borrowed access with explicit lifetime rules.
  • Measured inventories, no-shim migrations and parity gates.
  • EntityRef remains reporting-only.

Do not copy mechanically

  • A single ~15-declaration budget for every rich projection.
  • Append-only semantics for interactive placement and routing edits.
  • Free functions for stateful router/index algorithms.
  • Whole-operation atomicity for every routing workflow.
  • A universal aggregate, common entity base or property bag.
  • A generic stage graph before a real fourth domain needs it.
  • A compiled Board that becomes another owner of connectivity.
  • Flattening high-level Python into low-level C++ vocabulary.
Concrete exception: BoardRouter::connect is intentionally atomic, while escape intentionally preserves successful per-pad work when another pad fails. Circuit-style whole-call atomicity therefore cannot be a universal Board rule; the operation’s documented contract and tests decide.

A compression envelope, not an acceptance target

07 · Gold target

Authoring models above.
Compiled truth below.

The target has three planes. Each plane has a different job, and no plane is allowed to quietly acquire another plane’s meaning.

CompiledBoard guarantees

  • Every placed footprint definition is resolved and embedded.
  • Physical entities reference a minimal immutable logical dependency snapshot.
  • Source model and resolver inputs have deterministic provenance digests.
  • Validation, SVG, CPL, KiCad and fabrication fan out without external authoring libraries.
  • It cannot mutate or redefine Circuit connectivity.

CompiledBoard is not

  • A second editable Board model.
  • A Workspace containing arbitrary entities.
  • A new owner of nets or pin membership.
  • A cache whose staleness is silently tolerated.
  • A reason to move Project orchestration into C++.

Board completeness is not greenfield: the current Board already caches footprint-definition snapshots, the PCB writer embeds them, and the reader can round-trip a complete Board byte-identically with an empty library. The approved architecture turns that demonstrated capability into an explicit lifecycle invariant instead of repeating late resolver injection.

The current route-request overload of add_track belongs on BoardRouter. Footprint/placement lookup, pad resolution, geometry projection and ratsnest calculation are derived queries/services. Routing and spatial indexes may retain stateful object APIs; “free” is not a dogma.

Structural boundary: authoring Board may borrow Circuit and a resolver while it changes. Only successful compile_board yields the complete standalone physical handoff. Missing footprint data is a compile failure with typed diagnostics—not an exporter-time fallback.

Typed manifest record

struct ArtifactDescriptor {
  ArtifactId id;
  ArtifactRole role;  // model | view | render | report | adapter | asset | delivery
  std::string kind;
  std::string format_version;
  std::string media_type;
  RelativePath path;
  Sha256 content_digest;
  std::vector<ArtifactDependency> depends_on;
  ProducerInfo producer;
};

Project’s 18 current artifact kinds remain useful; role is an orthogonal lifecycle property. Dependencies and digests let a bundle loader reject stale viewer caches, mixed-build outputs or a compiled Board produced from different logical/physical inputs.

Recommended bundle layout

build/controller/ ├── manifest.volt.json ├── models/ │ ├── logical/controller.volt.logical.json │ ├── schematic/main.volt.schematic.json │ └── pcb/main.volt.pcb.json ├── views/ │ ├── schematic/main.volt.schematic-view.json │ └── pcb/main.volt.pcb-view.json ├── renders/ │ ├── schematic/main.svg │ └── pcb/main.svg ├── reports/ │ ├── diagnostics.json │ ├── tests.json │ ├── bom.json │ └── cpl.json ├── adapters/kicad/main.kicad_pcb └── assets/models3d/…

Manufacturing remains a separate profile-specific delivery package. Existing v1 bundle paths should remain readable until a deliberate manifest/version migration; this layout is the target shape, not permission for an unversioned break.

Owner-local codecs

logical::read/write_document, schematic::read/write_document and pcb::read/write_document sit beside their typed models. They preserve authoritative semantics and lifetime requirements.

Fan-out services

Render, BOM, CPL, KiCad and fabrication consume typed models or CompiledBoard. They are algorithms/adapters, not alternate models and not aggregate mutations.

12 · Verification

Evidence gates,
not confidence statements

The following gates convert the approved direction into independently verifiable work. A slice is not done merely because new APIs exist.

  • Public API snapshots and architecture checkers pin every accepted root/binding boundary.
  • Lexical inventories report zero production callers of removed paths.
  • Structural failures leave owner bytes unchanged where operations promise atomicity.
  • Board routing tests preserve both atomic connect and partial-success escape contracts.
  • Logical, Schematic and Board read/write/read plus write/read/write remain deterministic.
  • Compiled Board succeeds with an empty external footprint library and fails clearly when incomplete.
  • Compiled provenance changes when any logical, physical, footprint or capability input changes.
  • Old/new SVG, CPL, KiCad and native fabrication outputs are semantically equivalent before cutover.
  • Stale/missing viewer sidecars never change authoritative model loading.
  • Manifest loader rejects stale dependency digests and mixed-build artifacts.
  • All 18 artifact kinds receive one documented role and complete dependency edges.
  • All 113 public Python exports, typed exception families, codes and entity metadata remain compatible.
  • Binding lifetime/destruction tests cover logical owner retention and cross-design errors.
  • Board persistence/export proves independence from Python-only footprint caches after compilation.
  • 41 CLI tests and 46 Project Framework tests remain green.
  • Three staged Project examples and three fluent schematic-sugar examples run end to end.
  • Bundle-backed info/model/diagnostics demonstrate that the Python entrypoint is not imported.
  • CLI JSON schemas, selectors, exit codes and stdout isolation remain stable.
  • Full developer suite, docs, clang-tidy, sanitizers, coverage and supported-platform CI pass.

Parity means

Same meaning, diagnostics, ownership failures, serialization and output semantics at the new owner/artifact boundary.

Parity does not mean

Preserving every storage-shaped root method, hidden resolver fallback, mixed cache field, or private PyCircuit forwarding path.

13 · Boundaries

What is settled.
What focused ADRs must freeze.

Explicitly unchanged

  • Accepted Circuit aggregate ADR and append-only kernel semantics; the final boundary is 15 callable operations and 16 declarations.
  • Completed Circuit semantic, persistence, diagnostic, Python, CI and API/storage-compression work under #261 and #266.
  • Circuit alone owns logical components, pins, nets and pin-to-net membership.
  • Schematic owns presentation; Board owns physical implementation; neither defines the netlist.
  • Bad design intent remains diagnostic; structural invalidity remains a mutation/load boundary failure.
  • Canonical logical, schematic and PCB model documents remain separate and deterministic unless deliberately versioned.
  • Public volt.Design, fluent Schematic/Board authoring and typed Python exceptions remain compatible.
  • Python owns source execution, Project orchestration and developer-facing run status; C++ owns semantic artifact construction, codecs and verified reopening. BuildContext stays stable.
  • Public design→schematic→board stages stay fixed pending measured pain.
  • CLI stays orchestration-only and preserves JSON/stdout/exit-code contracts.
  • KiCad remains one-way interop; native fabrication remains manufacturing authority.
  • No kernel removal APIs, universal entity base, property bag or EntityRef mutation handles.

Approved direction and bounded ADR choices

  1. Name and lifetime: CompiledBoard is approved as an immutable artifact per named Board; D2 freezes its minimal logical dependency snapshot and owning loaded-project lifetime.
  2. Invalidation: eager recompile, explicit compile, or digest-keyed lazy cache?
  3. Board cardinality: multiple named Boards are approved. In v1 they are complete alternative physical implementations of one Design, not implicit assembly partitions.
  4. Schematic vocabulary: exact complete specs, update forms and whether SheetRegionId becomes strongly typed.
  5. Binding lifetime: explicit owner retention versus binding policy details.
  6. PCB versioning: early model/view split with v3 import or defer until bundle loading lands?
  7. Manifest evolution: role vocabulary, schema/version and checksum scope.
  8. Bundle API: a public C++ loader and typed native artifact graph are approved; D3 freezes exact types and read windows.
  9. CLI UX: shared --bundle flags or a new inspect command?
  10. Low-level Python: whether read-only native model inspection is justified later.
  11. Generic stages: the evidence threshold and first concrete domain that would admit StageKey[T].
  12. Delivery defaults: exports are opt-in; D3/Q4 freeze the default authoritative payload and legacy flat-output transition.

Approved direction; implementation remains gated

Option C is approved: owner-aligned Circuit/Schematic/named Board models, CompiledBoard, public C++ bundle reopening, a typed artifact graph and selected exports. Keep #269 open until focused ADRs record the remaining bounded choices and an admitted native GitHub dependency graph exists. This architecture reference directs those ADRs; it does not authorize implementation by itself.
