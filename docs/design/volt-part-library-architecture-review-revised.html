This revised page refines only the part-library seam. It preserves the original focused draft, does not replace the post-Circuit review, and is not the full refactor plan.
01 · Target decision

One declaration per exact part. One native path.

Adopt a first-class, C++-owned part-library system with a deliberately small primary vocabulary and a typed semantic substrate that can grow. Python remains the pleasant declaration surface; native definitions, validation and compiled artifacts remain the portable meaning.

Gold target: author a shared contract once, publish each exact part once, select it by digest.

ComponentDefinition owns the reusable logical contract and contains its PinDefinitions and typed feature bindings. PartDefinition is one exact manufacturer implementation of that contract. ComponentInstance is one occurrence in a design and records its chosen part reference. These are three lifecycle roles, not three ways a library author must describe the same device.

01 / DEFINE

Declarative source

Python or C++ declarations, generators, assets and ordinary unit tests.

02 / COMPILE

Native services

C++ validates, hashes, queries, resolves and emits the library bundle.

03 / CONSUME

Immutable closure

Projects snapshot only the selected parts and assets required by their builds.

Normal authoring declares each meaning once. A reusable component contract is declared once; every exact part is then declared once against it. A convenience builder may nest a genuinely unique contract inside one exact-part declaration, but the explicit examples below keep the two lifecycle roles visible.
Closed canonical core; open feature composition. Libraries may compose versioned features around zero, one, two or many logical pins, but any meaning required by core ERC must lower to the same kernel-owned subject, observable and constraint semantics. The first implementation slice supports only canonical Voltage and Current; later quantities extend the substrate without changing the three primary concepts.
Reading key. Section 02 is NOW: evidence from exact main. Sections 01 and 03–12 are the TARGET proposed for approval. Target nouns and APIs are intentionally concrete, but they are not claims about current code or accepted ADRs.
Companion relationship. The broader post-Circuit review remains the owner of the three-plane project architecture, multiple Boards, CompiledBoard, typed artifact graph, selected exports and C++-first bundle API. This review supplies the reusable-part boundary. A separate full refactor plan follows only after this focused model is approved.

03 · Target three-concept model

The vocabulary is smaller than the implementation.

Only three concepts belong in the primary mental model. Pins are accurate, first-class kernel data, but they are contained by the component contract rather than presented as another way to define a part.

Reusable logical contract

ComponentDefinition

Defines what the device is electrically: its logical interface, features every implementation must satisfy, allowed or required instance-input shapes, and reusable relationships across terminals.

Exact implementation

PartDefinition

One manufacturer + MPN that implements the component contract with exact canonical constraints and characteristics, per-pin physical mapping, provenance and required assets.

PartDefinition — implements exactly one → ComponentDefinition. ComponentInstance is an occurrence of that component and may select the exact part. Shared contracts and exact implementations are each authored once; optional nested source syntax does not create a second persisted ownership model.

Supporting machinery, not more meanings of “part”

A

PinDefinition

nested contract data

Genuine kernel data because every connected PinInstance needs stable reusable identity, semantics and diagnostic context. Exact package-pin numbers and tighter per-pin limits are supplied by the implementing part, keyed to the logical pin.

B

Typed feature graph

nested semantic data

A feature binds a versioned schema to named logical-pin roles and may cover the whole part, one pin, a directed path, a supply domain or a pin group. Its exact implementation emits canonical constraints and typed extension characteristics. Features group meaning; they do not redefine Voltage, Current or other kernel observables.

C

Projection attachments

symbol · footprint · 3D · simulation

The default logical schematic symbol belongs to ComponentDefinition. Exact footprint, package mapping, 3D and simulation attachments belong to PartDefinition; an exact-part symbol variant is an explicit override, never a second default owner.

D

PartLibraryBundle

packaging

The immutable published closure of definitions, references, required assets and evidence. It is distribution, not a fourth domain entity.

E

ModuleDefinition

separate reusable circuit

A multi-component circuit or reference design. It consumes components and parts; it is intentionally outside the atomic part vocabulary.

Manufacturing readiness stays visible. An unresolved ComponentDefinition is allowed during logical exploration. It must fail Board/BOM readiness until a compatible exact PartDefinition is pinned. A manufacturable part never means “some resistor in this family”.

05 · Target electrical semantics

Closed canonical core. Open feature composition.

A feature is an ergonomic grouping, not a private electrical dialect. Every value needed by core selection or ERC lowers to one kernel-owned subject, observable, meaning, value and condition set; exact parts and custom libraries cannot redefine those semantics with near-synonym keys.

Authoring → native build → canonical bundle → native reopen → ERC must preserve and query the same semantic records.

Unknown values stay unknown. Core validation never infers electrical meaning from a feature name, pin label, category string or Python keyword.

Selection requirements are not copied ratings. An unresolved instance may carry a requirement used to find a part. Direct design.add(exact_part) does not restate intrinsic values. Any requirement retained after selection remains independently authored design intent checked against the referenced part.
Physical identity needs two explicit seams. PinDefinition owns a stable logical PinKey. The exact PartDefinition owns PinKey → [PackageTerminalKey], then PackageTerminalKey → [FootprintPadKey] for its selected footprint; the footprint artifact owns pad definitions and geometry. Same-labelled terminals and pads may be authoring shorthand, never stored identity. Current PinDefinition.number can mean logical or package number, so migration must separate those meanings.

One generic record shape; two canonical observables first

// Illustrative native substrate. Names are not accepted API.
ElectricalRecord {
  subject:    ElectricalSubjectRef,
  observable: CanonicalObservableId,
  meaning:    Characteristic | AcceptedRange | ProvidedRange |
              AbsoluteLimit | Requirement | Capability,
  value:      Quantity | Range | Tolerance,
  conditions: ConditionSet,
  evidence:   optional EvidenceRef
}

FeatureBinding {
  key:       FeatureKey,
  schema:    FeatureSchemaRef,
  terminals: named role → [PinKey],
  relations: optional named relation refs
}
// The first slice admits only electrically framed subjects.
ElectricalSubjectRef =
    FramedPin       { pin, reference }
  | DirectedRelation { relation, positive_role, negative_role }
  | SupplyDomain   { positive_pins, return_pins };

// Bare "voltage on pin P" and unsigned "current through path X"
// are invalid records, not values for ERC to guess about.

The first vertical slice registers only canonical Voltage and Current. That is intentionally narrow: it proves versioned semantic identity, units, explicit reference and direction, conditions, feature mapping, serialization, query and ERC consumption before adding Power, Time/Frequency, Temperature, R/L/C or other observables. Later observables may admit additional subject shapes only when their native schema defines unambiguous semantics.

LayerExtension ruleExampleCore ERC treatment
Canonical electrical semanticsClosed, versioned and C++-owned.Voltage, Current, accepted range, provided range, requirement, capability, absolute limit.Queried directly; feature names are irrelevant.
Feature schemasOpen and composable, but standard meaning must map to canonical records.Supply consumer/source, diode junction, digital-I/O bank.Consumes the emitted canonical records and schema conformance.

Feature conformance is a native join, not convention. The builder substitutes each FeatureBinding’s terminal and relation roles into its schema’s expected canonical subjects, then matches exact-part records by stable (subject, observable, meaning, condition-key). Typed extension values persist their (FeatureSchemaId, CharacteristicId). Dangling or duplicate singular bindings reject structurally; missing schema-required records reject an implementation, while optional catalogue completeness remains diagnosable and may be promoted to an official publish gate.

Voltage and Current already exercise the whole architecture

// Exact ESP32 implementation records.
subject main_supply = domain(positive=[pin("3V3")],
                             returns=[pin("GND")]);

record(main_supply, Voltage, AcceptedRange, 3.0 V .. 3.6 V);
record(main_supply, Voltage, AbsoluteLimit, -0.3 V .. 3.6 V);
record(main_supply, Current, Requirement, minimum_source_capacity(0.5 A));

// Exact regulator output records.
record(regulator_output, Voltage, ProvidedRange, 3.2 V .. 3.4 V);
record(regulator_output, Current, Capability, continuous(0.6 A));

// ERC compares canonical records, not component classes:
provided_voltage ⊆ accepted_voltage
provided_current ≥ required_current
Relations carry orientation; instances project them onto Circuit connectivity. A directed relation owns its terminal-role order, so voltage polarity, signed current and forward/reverse qualifiers are interpreted relative to that order. At instantiation, C++ maps the relation’s local PinKeys through the ComponentInstance pins to Circuit nets and domains. ERC compares resolved source and consumer records there; no Python helper or feature name participates in the result.
A feature helper may use friendly vocabulary; the artifact may not.

A custom helper may accept vin_range, but it must serialize the canonical accepted Voltage range on a referenced supply domain. A second custom key such as input_supply does not satisfy ERC readiness. Conflicting canonical records remain visible and diagnosable rather than silently choosing one.

Always kernel data

Stable subjects, canonical observable IDs, meanings, dimensioned values, operating conditions, feature bindings, exact identity, mappings and provenance.

Separate attachment

Vendor or approximate simulation models, 3D interchange, measured evidence and analysis-specific parameter sets.

Never hidden meaning

Free-form .facts(**kwargs), Python callbacks, name inference, silently aliased synonyms or solved simulation outputs copied into the part.

Current gap, first migration priority: exact main has typed scalar/range/tolerance storage but no stable subject/feature binding, operating-condition model or canonical cross-pin/domain semantics. ComponentDefinition and PartDefinition also lack the proposed native implementation relationship. Packaging the existing maps first would only make the semantic split portable.
06 · Target simulation boundary

Simulation-ready, not simulation-specific.

Typed ratings are not a simulator model. The catalogue should preserve both without coupling Volt’s canonical meaning to SPICE, IBIS, an internal solver or a Python callback.

Canonical inputsCircuit connectivity + part characteristics and constraints
→
Simulation serviceconsumes optional model attachment + hashed backend payloads
→
Analysis artifactSimulationGraph + SimulationResult

Model attachment owns

  • model kind and supported analyses;
  • terminal schema and logical PinKey bindings;
  • typed parameter bindings and corners;
  • fidelity: ideal, approximate, vendor or measured;
  • limitations, provenance, licence and content digest;
  • backend payload refs such as SPICE, IBIS or S-parameter files.

Readiness diagnostics own

  • no model for the requested analysis;
  • model terminals or units do not line up;
  • required part attributes are missing;
  • backend cannot represent the chosen model;
  • model fidelity or ignored effects are insufficient;
  • PCB parasitics are absent when a case expects them.

Scope line: this review reserves the correct attachment boundary; it does not implement a solver, SPICE integration or simulation API. Absence of a model cannot invalidate an otherwise useful schematic/PCB/BOM part. It only affects validate_for_simulation(requested_analysis).

Simulation reduces uncertainty; it does not prove a physical product. Hardware validation and manufacturing evidence remain the loop-closing proof, and their attestations must state exactly what was tested.

Published library bundle includes

  • canonical definitions and schemas;
  • every referenced projection and actual asset byte;
  • licence and provenance needed to consume them;
  • compact verification state keyed to content hashes;
  • dependency and compatibility metadata.

Published library bundle excludes

  • source Python and test dependencies;
  • every render, screenshot, CI log and oracle diff;
  • live stock, price and distributor offers;
  • generated CAD tooling or regenerated model output;
  • project-specific sourcing snapshots and compiled outputs.

These are two different closures. A PartLibraryBundle contains the whole published catalogue and its referenced assets. A project snapshot begins at each selected PartDefinition, follows the immutable reference to its ComponentDefinition, and vendors only those transitive closures. Updating a source dependency creates new compiled outputs; previous CompiledBoard artifacts remain untouched.

Sourcing stays separate. Manufacturer identity and approved exact alternates belong to the catalogue. Distributor SKU, price, stock and quote time remain project-owned sourcing snapshots, so market churn cannot change a canonical part hash.
10 · Target verification contract

“Testable on its own” is a publish contract.

Normal pytest/Catch2 tests remain transparent author-authored evidence. Native gates prove that the shipped artifact, not merely the source declarations, carries the promised meaning.

Deterministic build

Build source twice in clean environments; manifests, canonical entries and asset digests match byte-for-byte.

gate · digest equality

Native reopen and round-trip

Open the emitted bundle through public C++, preserve each part → component relationship, feature binding and canonical electrical record, then serialize equivalent meaning without Python.

gate · no authoring runtime

Structural integrity

Reject missing component contracts, malformed units, duplicate keys, dangling logical-pin refs, bad logical-pin/package-terminal or terminal/pad bindings, missing asset bytes and digest mismatches.

gate · invalid state impossible

Contract implementation

Prove every exact part implements one existing component contract; each feature satisfies its terminal roles; every subject and package mapping resolves to stable logical pins or relations.

Canonical interoperability

Voltage and Current ERC

Publishable versus loadable. Malformed artifacts fail at the native read boundary. Incomplete but structurally valid parts stay inspectable and emit diagnostics. 11 · Target focused implementation order

Ten bounded slices. Prove Voltage and Current first.

This is the implementation order for part libraries, not the later full-project refactor plan. Establish the generic subject/feature/record substrate and one useful ERC vertical before packaging today’s dual semantics.

00

Freeze current dual behaviour

Snapshot the three explicit routes—Design.define_component, legacy Library.component and modern Library.part/Part—plus built-in helpers, diagnostics, hashes and v4 goldens.

Stop gate: exact-main inventories and reproducible fixtures expose both current lowering paths.
01

Land the semantic substrate with V/I

Add stable logical PinKeys, explicitly framed pin, directed-relation and supply-domain subjects, versioned feature schemas and canonical ElectricalRecords. Implement only Voltage and Current with meanings, conditions and native query.

Stop gate: malformed schemas/records reject; standard and custom helpers reopen as identical canonical V/I meaning through public C++.
02

Make exact parts implement, not duplicate

Add the immutable component-contract reference; replace copied PartPin(PinDefinition, attributes) meaning with feature implementations, canonical exact records, logical-pin → package-terminal bindings and terminal → footprint-pad bindings.

Stop gate: native reopen proves part → component identity, feature conformance, complete V/I records and valid physical mappings.
03

Pin exact selection and prove canonical ERC

Add native LibraryPartRef, validate contract compatibility and resolve intrinsic part records rather than copying PhysicalPart truth. Project each part-local framed pin, directed relation and supply domain through instance pins onto Circuit connectivity. Route Python through this C++ operation.

Stop gate: exact regulator/source → ESP32 and LED fixtures emit under/overvoltage plus insufficient and aggregated continuous-current diagnostics through one native path, before library packaging begins.
04

Add the native library aggregate

Introduce PartLibraryBuilder, manifest identity, stable keys, duplicate checks, public query and explicit exact resolution.

Stop gate: C++ alone builds, validates, queries and instantiates an in-memory catalogue.
05

Make asset closure and bundle real

Make symbols and footprints loadable artifacts; hash actual GLB/STEP and optional simulation bytes; implement deterministic PartLibraryBundle codecs.

Stop gate: missing bytes reject and source build/prebuilt reopen preserve every entry and asset digest.
06

Lock and snapshot project dependencies

Record library/version/key/digests automatically and vendor only the selected part → component → asset closure into ProjectBundle and compiled handoffs.

Stop gate: the project reopens offline; earlier CompiledBoards remain byte-stable after library upgrades.
07

Add standalone CLI and package workflow

Deliver build, check, test, inspect and render over source and prebuilt bundles; support pinned path/Git dependencies and volt.lock.

Stop gate: inspection never executes authoring Python and CLI error/JSON contracts are stable.
08

Dogfood, then retire duplicate routes

Expand the early V/I proof through real shipped definitions, then migrate all 17 STM32 entries and 11 manual example selections before deprecating copied selection and ad hoc lowering.

Stop gate: canonical ERC, real examples, bundles, Boards, BOM/CPL and renders pass before any deletion.
09

Pilot volt-standard-parts

Publish a small high-confidence catalogue limited to connectors, LEDs/diodes, regulators and supply parts, selected MCUs, and packaged modules whose publish contract is expressible with the V/I slice. Defer passives that require canonical R/L/C values.

Stop gate: every pilot part is native-loadable, licensed, deterministic, board/BOM ready, rendered and independently reviewed.
Expansion follows evidence. Power, Time/Frequency, Temperature, R/L/C and richer value shapes are later observable slices over the same subject/feature/record substrate; their order is decided from concrete diagnostics. Simulation then attaches to stable pins, relations and characteristics rather than becoming the canonical part model.
12 · Target decisions and boundaries

The recommendation is concrete. Approval and implementation details remain open.

The proposed user-level architecture no longer needs an A/B/C choice. If approved, remaining decisions can be resolved per bounded slice without reopening the three-concept ownership model.

Proposed for approval

  • Exactly three primary concepts: ComponentDefinition, exact PartDefinition and ComponentInstance.
  • PinDefinition[] is nested logical-contract data, not a fourth meaning of “part”.
  • The nested typed feature graph is supporting semantic machinery, not another primary part concept.
  • One exact part implements one component contract; one instance selects it through one integrity-bearing exact ref.
  • There is no separate selected-occurrence owner and no permanent copy of intrinsic part truth on each instance.
  • Assets, models, bundles and results are attachments, packaging or outputs—not peer part concepts.
  • The default logical symbol belongs to the component contract; exact footprint, package mapping, 3D and simulation attachments belong to the exact part.
  • Visible authoring, service and artifact planes.
  • C++ owns part meaning, validation, codecs, loading, query, resolution and artifact construction.
  • Python is a concise declaration and typed-handle surface.
  • Closed canonical electrical semantics and open feature composition: helpers may vary, but ERC consumes one native language.
  • The first canonical observable slice is Voltage and Current on explicitly framed pins, directed relations and declared supply domains; bare pin voltage and unsigned path current are invalid.
  • Every canonical record carries subject, observable, meaning, typed value and optional operating conditions/evidence.
  • Free-form .facts(**kwargs) and Python-owned electrical callbacks are not canonical part APIs.
  • Namespaced extension characteristics remain preserved but do not satisfy core ERC unless explicitly mapped.
  • Optional simulation models are separate hashed projections.
  • Queries return candidates; persisted design state pins an exact part digest.
  • Personal/team and official libraries share one format and test contract.
  • Local/Git dependencies first; registry later and outside the kernel.
  • Projects snapshot only consumed parts and assets.
  • GLB is viewer-ready; STEP is optional interchange.
  • Verification attestations key to immutable part hashes.

Still open, with recommendation

  1. Contract granularity: use semantic/pinout compatibility contracts, not one universal “regulator” or “MCU” contract.
  2. Exact physical-pin storage: key logical-pin → package-terminal mapping by stable PinKey, then bind package terminals explicitly to footprint pads; keep electrical constraints on framed pin, directed relation and domain subjects and do not expose PartPin as peer vocabulary.
  3. Portable component identity: immutable content digest inside bundles, with a readable stable key alongside it.
  4. Contract evolution: begin with exact component-contract identity; define any v1→v2 compatibility or adapter mechanism only with explicit, testable semantics.
  5. One part, multiple contracts: support one canonical contract initially; defer explicit adapters until a real case proves the need.
  6. Initial meaning set: finalize the v1 meanings and merge/cardinality rules for accepted/provided ranges, characteristics, requirements, capabilities and absolute limits.
  7. Conditions and expressions: begin with literal Voltage/Current conditions and bounded reference/scale expressions; do not accept arbitrary formulas or Python callbacks.
  8. Conflicts: define when multiple conditioned records coexist, when compatible constraints intersect and when contradictory single-valued meaning blocks publication.
  9. Current aggregation: begin with compatible continuous-current requirements in one resolved supply domain; define peak, duty-cycle, operating-mode and mutually exclusive load semantics before aggregating those records.
  10. Next observables: choose Power, Time/Frequency, Temperature and R/L/C order only from proven ERC, selection or simulation-readiness use cases.
  11. Instance overrides: define a narrow typed allowlist for occurrence intent; intrinsic MPN characteristics can never drift.
  12. Physical bundle: deterministic archive for distribution plus exploded developer inspection.
  13. volt.part schema migration: decide v5 versus another explicit successor, retain v4 reading during migration, and provide deterministic old-artifact conversion.
  14. Human and integrity versions: semver names releases for people; immutable content digests identify every component contract, part and asset.
  15. Python exports and resolution: typed names plus explicit discovery; path/Git/package dependencies in volt.toml, exact digests in volt.lock.
  16. Alternates: refs to separately defined exact parts, not unverified bare MPN strings.
  17. Simulation and official scope: design the first licensed payload set separately; begin the official catalogue as a small evidence-rich pilot.

Explicitly unchanged

A

Kernel ownership

Circuit remains append-only logical owner. Schematic and Board remain presentation and physical projections; neither creates or reinterprets connectivity.

B

Part selection

Exact selected-part intent remains logical/design-owned, not Board-owned. BOM/CPL and manufacturing project from it.

C

Project architecture

One project may produce multiple named Boards. Each CompiledBoard is an immutable historical artifact keyed by its inputs.

D

Export posture

KiCad and other outputs remain selected adapters. Native artifacts stay authoritative; no exporter or viewer becomes an alternate part owner.

E

Runtime discipline

No CAD generator executes during ordinary project consumption. No registry, live distributor service or package manager enters the kernel.

F

Change control

This page is a proposal and companion review. It changes no accepted ADR, product code, issue state or GitHub state.

G

Full refactor plan

Not created in this step. It waits for approval of this focused part-library model, then becomes a separate cross-project plan for chief-of-staff handoff.
